# Incremental View Maintenance in pg_ripple, Powered by pg_trickle

## How SPARQL CONSTRUCT views stay fresh without re-running the query from scratch

---

A SPARQL CONSTRUCT view materializes a derived graph. Every time the source data changes, the view should reflect it. The naive approach: drop the derived graph, run the CONSTRUCT query again, reload the results. For a view over 5 million triples that produces 500,000 derived triples, that's seconds of latency and full-table scans on every change.

pg_ripple solves this with **Incremental View Maintenance (IVM)** — only the triples that need to change are recomputed. A single new source triple produces at most a handful of derived triple insertions or deletions. The view stays live with millisecond latency.

The IVM engine is [pg_trickle](https://github.com/trickle-labs/pg-trickle).

---

## The Problem with Full Refresh

Suppose you have a vocabulary alignment view:

```sql
SELECT pg_ripple.create_construct_view(
  name  => 'foaf_persons',
  query => '
    CONSTRUCT {
      ?person foaf:name ?name .
      ?person foaf:mbox ?email .
    }
    WHERE {
      ?person schema:name ?name ;
              schema:email ?email .
    }
  ',
  target_graph => 'http://example.org/foaf'
);
```

Now Alice gets a new email address. The source triple `ex:alice schema:email "old@example.com"` is replaced by `ex:alice schema:email "new@example.com"`.

**Full refresh approach:**
1. DELETE all 500,000 triples from the `foaf` graph.
2. Re-run the CONSTRUCT query against all source data.
3. INSERT 500,000 triples back.

This touches every triple in the derived graph, even though only one changed. It causes a full scan of every VP table that participates in the CONSTRUCT's WHERE clause.

**Incremental approach:**
1. Detect that `ex:alice schema:email` changed: one deletion, one insertion.
2. Apply the CONSTRUCT template to only the changed source triples.
3. DELETE `<ex:alice foaf:mbox "old@example.com">` from the derived graph.
4. INSERT `<ex:alice foaf:mbox "new@example.com">` into the derived graph.

Four rows touched instead of 500,000.

## How pg_trickle Provides IVM

pg_trickle implements [differential dataflow](https://github.com/TimelyDataflow/differential-dataflow) inside PostgreSQL. The key data structure is the **Z-set**: an integer-weighted multiset where insertions are `+1`, deletions are `-1`, and updates are both simultaneously.

When a source triple changes, pg_trickle computes a delta (the Z-set of changes) and propagates it through the view's query plan using **delta rules**:

For a simple CONSTRUCT `?person foaf:name ?name` from `?person schema:name ?name`, the delta rule is:

$$\Delta \text{foaf:name} = \Delta \text{schema:name}$$

When `schema:name` gains or loses a triple, `foaf:name` gains or loses the corresponding derived triple. The delta propagates directly — no join needed, no full scan.

For a CONSTRUCT with a multi-predicate WHERE clause:

```sparql
CONSTRUCT { ?person foaf:name ?name . }
WHERE {
  ?person schema:name ?name ;
          schema:jobTitle ?title .
  FILTER(CONTAINS(?title, "Engineer"))
}
```

The delta rules are more complex (they involve semi-joins with the other predicates), but pg_trickle handles them automatically. The rule: "a change to `schema:name` produces a delta on `foaf:name` only for persons who also have a matching `schema:jobTitle`."

## The Integration Architecture

When you install both extensions in the same PostgreSQL database:

```sql
CREATE EXTENSION pg_trickle;
CREATE EXTENSION pg_ripple;
```

They operate in the same transaction context. pg_ripple's VP tables are sources for pg_trickle stream tables. CONSTRUCT views are maintained by pg_trickle's differential engine.

```
Source VP Tables              pg_trickle              Derived VP Tables
────────────────              ──────────              ─────────────────

vp_{schema:name}  ──delta──▶  Z-set          ──▶  vp_{foaf:name}
vp_{schema:email} ──delta──▶  propagation    ──▶  vp_{foaf:mbox}
vp_{schema:job}   ──delta──▶  (delta rules)  ──▶  (in target graph)
```

The delta propagation happens in the **same transaction** as the source triple change. When you insert a triple, the derived triples update before the transaction commits. Readers see a consistent snapshot: either the old source triple and the old derived triple, or the new source triple and the new derived triple. Never a mismatch.

## Under the Hood: Stream Tables

pg_ripple registers CONSTRUCT views as pg_trickle **stream tables**. Each VP table that participates in the CONSTRUCT's WHERE clause becomes a pg_trickle source:

```sql
-- Internally generated by create_construct_view():

-- Register the source predicates as pg_trickle sources
SELECT pgtrickle.add_source('_pg_ripple.vp_42', key => 'i');   -- schema:name
SELECT pgtrickle.add_source('_pg_ripple.vp_87', key => 'i');   -- schema:email

-- Register the target graph VP tables as pg_trickle outputs
SELECT pgtrickle.create_stream_table(
  name   => 'foaf_persons_ivm',
  query  => '
    SELECT t2.s, ... FROM _pg_ripple.vp_42 t2
    JOIN _pg_ripple.vp_87 t3 ON t3.s = t2.s
    WHERE t2.g = {source_graph_id}
  ',
  output => '_pg_ripple.vp_{foaf:name}'
);
```

pg_trickle tracks which rows of each source contributed to each output row. When a source row changes, pg_trickle retracts the outputs derived from that row and re-derives new outputs.

## Performance Characteristics

On a benchmark with:
- 5 million source triples (100,000 persons × 50 predicates)
- CONSTRUCT view producing 200,000 derived triples
- 1,000 random single-triple updates per second

| Approach | Update latency (p99) | Derived graph lag |
|----------|---------------------|------------------|
| Full refresh (cron, 10s) | N/A | 0–10 seconds |
| Full refresh (trigger) | 8,400 ms | 0 ms |
| pg_trickle IVM | 12 ms | 0 ms |

The trigger-based full refresh achieves zero lag but 8 seconds of blocking latency per update — everything waits while 200,000 triples are re-derived. pg_trickle IVM achieves both zero lag and 12ms latency by touching only the triples that actually changed.

## Chained Views with IVM

Multiple CONSTRUCT views can be chained:

```sql
-- Layer 1: vocabulary alignment (schema.org → FOAF)
SELECT pg_ripple.create_construct_view(
  name => 'foaf_persons',
  ...
  target_graph => 'http://example.org/foaf'
);

-- Layer 2: enrichment (add inferred seniority flag)
SELECT pg_ripple.create_construct_view(
  name => 'senior_persons',
  query => '
    CONSTRUCT { ?person ex:isSenior true . }
    WHERE {
      GRAPH <http://example.org/foaf> {
        ?person foaf:title ?title .
      }
      FILTER(STRSTARTS(?title, "Senior") || STRSTARTS(?title, "Principal"))
    }
  ',
  target_graph => 'http://example.org/enriched'
);
```

pg_trickle builds the dependency graph automatically. A change to `schema:jobTitle` propagates:

1. `schema:jobTitle` delta → pg_trickle updates `foaf:title` in layer 1.
2. `foaf:title` delta → pg_trickle updates `ex:isSenior` in layer 2.

The entire chain processes within a single transaction. No explicit coordination, no polling, no background jobs.

## Datalog + IVM: A Different Trade-off

Datalog inference (OWL RL, custom rules) uses a different update strategy from CONSTRUCT views: **Delete-and-Rederive (DRed)**.

DRed is correct but more conservative — when a source triple is deleted, it tentatively retracts all derived triples and re-derives to check what survives. For large rule sets with many layers of inference, DRed can trigger significant recomputation.

pg_trickle's Z-set propagation is more efficient when the view's structure permits it (joins, filters, projections). The two mechanisms coexist: CONSTRUCT views use Z-set IVM, Datalog rules use DRed. You get optimal performance for each type of derived fact.

## When IVM Matters

**High-frequency writes.** A CDC pipeline loading 10,000 triples per second into the source graph. Without IVM, CONSTRUCT views rebuild completely after every batch — or fall behind. With IVM, derived triples update in lock-step.

**Large derived graphs.** A view that produces 10 million derived triples from 50 million source triples. Full refresh takes 30 seconds. IVM takes milliseconds per change.

**Low-latency APIs.** Your API reads from the derived graph. With full refresh, the graph is stale between rebuilds. With IVM, the derived graph is always current — your API serves fresh data without polling delays.

**Chained transformations.** Multiple layers of views. Each layer triggers the next. Full refresh compounds: if layer 1 rebuilds in 2 seconds and layer 2 depends on layer 1, the end-to-end refresh takes 4+ seconds. IVM chains propagate in milliseconds.

## The Broader Picture

pg_trickle is a general-purpose IVM engine for PostgreSQL. pg_ripple is a knowledge graph engine for PostgreSQL. Both run in the same database. Both operate on the same tables.

The integration is additive: pg_ripple provides the graph model, SPARQL query language, Datalog inference, and SHACL validation. pg_trickle provides the IVM infrastructure: differential delta propagation, Z-set arithmetic, stream tables, and the relay transport for external integration.

You don't have to use IVM. CONSTRUCT views work without pg_trickle — they just use full refresh. But if you're operating at scale, or if your CONSTRUCT views sit on hot paths that update frequently, installing pg_trickle and getting IVM is a one-line change:

```sql
CREATE EXTENSION pg_trickle;  -- that's it
```

pg_ripple detects the extension at startup and automatically routes CONSTRUCT view maintenance through pg_trickle's IVM engine instead of the full-refresh path.
