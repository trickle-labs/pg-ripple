## Release theme

v0.64.0 is a truth-in-release milestone. It turns the PLAN_OVERALL_ASSESSMENT_8 process findings into CI-enforced release policy: public claims must match implemented code paths, optional features must expose whether they are active or degraded, and release artifacts must be built from immutable dependencies. The release lays the operational foundation for the v0.65-v0.67 remediation sequence.

## Deliverables

### TRUTH-01 - Feature status SQL API

Add `pg_ripple.feature_status()` returning one row per major capability:

```sql
RETURNS TABLE(
    feature_name TEXT,
    status TEXT,
    dependency TEXT,
    degraded_reason TEXT,
    ci_gate TEXT,
    docs_path TEXT,
    evidence_path TEXT
)
```

Allowed `status` values:

- `implemented` - normal execution path is wired, tested, and documented.
- `experimental` - available behind a GUC/feature flag with documented limits.
- `planner_hint` - optimization guidance exists but is not a custom executor/operator.
- `manual_refresh` - feature is correct only when a manual refresh is invoked.
- `stub` - API exists but production behavior is not implemented.
- `degraded` - dependency or configuration is missing; fallback behavior is active.
- `planned` - roadmap item exists, no user-facing implementation.

Initial rows must include at least:

| feature | required status after v0.64.0 |
|---|---|
| `construct_writeback` | `manual_refresh` until v0.65.0 closes incremental maintenance |
| `arrow_flight` | `stub` until v0.66.0 delivers Arrow IPC streaming |
| `wcoj` | `planner_hint` unless a true triejoin executor is implemented |
| `shacl_sparql_constraint` | `implemented` |
| `shacl_sparql_rule` | `planned` or `degraded` until routed through the fixed derivation engine |
| `citus_service_pruning` | `planned` or `experimental` until translator-integrated |
| `citus_hll_distinct` | `planned` until aggregate SQL generation emits HLL |
| `sparql_cursor_streaming` | `planned` until it avoids full materialization |

Verification:

- New pg_regress test `feature_status.sql` asserts all known partial features report honest status values.
- `feature_status()` must not silently report `implemented` unless a named CI gate exists.

### TRUTH-02 - Deep `/ready` readiness in `pg_ripple_http`

Extend `/ready` to return HTTP 200 only when required dependencies are healthy and no required feature is degraded. Optional features appear in a `features` object with the same statuses as `pg_ripple.feature_status()`.

Required response fields:

- PostgreSQL connectivity and extension version.
- Migration version vs control-file version.
- Feature status snapshot.
- Release evidence artifact hash, if present.
- Degraded feature reasons.

Verification:

- HTTP integration test with default configuration.
- Test that missing optional dependencies are reported without failing readiness.
- Test that required broken dependencies fail readiness.

### TRUTH-03 - Immutable GitHub Actions

Replace every third-party `uses:` reference with a full 40-character commit SHA. Add `scripts/check_github_actions_pinned.sh` and a CI step that fails when any workflow contains a mutable ref such as `@v6`, `@main`, `@stable`, or a branch name.

Verification:

- CI lint reports zero mutable action refs.
- Dependabot or an equivalent update guide documents how pinned actions are refreshed.

### TRUTH-04 - Release workflow digest integrity

Remove `continue-on-error: true` from Docker build/push. Capture the image digest from `docker/build-push-action`, scan that immutable digest with Trivy, and fail the release if the scanned digest differs from the pushed digest.

Verification:

- Simulated build failure fails the release job.
- Trivy scan uses `ghcr.io/trickle-labs/pg-ripple@sha256:...`, not a mutable tag.
- Release evidence dashboard records the digest, SBOM file, and scan result.

### TRUTH-05 - Documentation truth pass

Correct public documentation to match the implementation status from PLAN_OVERALL_ASSESSMENT_8:

- README current-version section must reflect v0.63.0 metadata or a generated version variable.
- Arrow Flight must be documented as a ticket and JSON stub until v0.66.0.
- WCOJ must be documented as cyclic BGP planner hints, not a true Leapfrog Triejoin executor, unless v0.66.0 implements a real executor.
- SHACL-SPARQL constraints must be distinguished from `sh:SPARQLRule`; SPARQL rules are not executed until v0.65.0/v0.66.0 routes them through the derivation kernel.
- `plans/implementation_plan.md` must use pgrx 0.18, matching `Cargo.toml`.
- v0.63 Citus function signatures in CHANGELOG/docs must match actual pgrx APIs.
- Add a top-level "Known limitations in the current release" section.

Verification:

- Docs/API signature drift check passes.
- README examples execute against a temporary database or are explicitly marked pseudo-code.

### TRUTH-06 - Roadmap status taxonomy and release evidence links

Every new roadmap item must include status and acceptance evidence:

| Status | Required evidence |
|---|---|
| Implemented | Code path wired into normal execution, behavior test, docs updated |
| Experimental | Feature flag/GUC, warning docs, limited test |
| Stub | Publicly labeled as stub; no production claims |
| CI-gated | Dedicated CI acceptance test or benchmark gate |
| Planned | Roadmap only; not described as released |

Verification:

- New script rejects roadmap/changelog phrases such as "implemented" or "delivered" when no evidence link exists.
- Release notes generation includes evidence links.

## Conflicts and risks

- This release deliberately changes public wording for previously released features. That can feel like a downgrade, but it is required to restore trust.
- `feature_status()` introduces a new SQL API and may require a migration if backed by a catalog table. A pure computed function avoids schema churn.
- Action SHA pinning requires maintenance; add Dependabot or a documented update process.
- Docs/API signature checks can be noisy at first. Gate only high-confidence mismatches initially, then tighten.

## Exit criteria

- Zero mutable GitHub Actions refs in all workflows.
- Docker release build/push failures are fatal, and Trivy scans immutable digests.
- `pg_ripple.feature_status()` reports honest statuses for all partial features identified in PLAN_OVERALL_ASSESSMENT_8.
- README, CHANGELOG, docs, roadmap, and implementation plan no longer overstate Arrow Flight, WCOJ, SHACL-SPARQL rules, Citus v0.63, or CONSTRUCT writeback.
- `just assess-release` passes locally and in CI.
- Release evidence dashboard artifact is generated in CI.
