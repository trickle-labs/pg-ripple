# v0.103.0 — Conflict Detection

**Theme**: Find contradictory rules before they cause problems.

> **In plain language:** When you have many reasoning rules, some may contradict each other —
> one rule concludes "this person is eligible for the loan" and another concludes "this person
> is ineligible". Catching these contradictions *after* they cause wrong decisions is expensive
> and embarrassing. This release adds two modes of conflict detection: a fast *static* analysis
> that checks rules for structural contradictions before you even run inference (like a compiler
> warning), and a *runtime* check that scans the live triple store for already-derived facts
> that violate your SHACL constraints. Both modes produce a structured report showing exactly
> which rules conflict and why.
>
> **Foundation document**: [plans/expert-system.md](../plans/expert-system.md)
>
> **Effort estimate: 2–3 person-weeks**

## Prerequisites

- v0.100.0 released and all tests passing (`_pg_ripple.derivations` table available for runtime mode; `_pg_ripple.rules` table with `head_pred` column available).
- SHACL `sh:disjoint` and `sh:not` constraints implemented (shipped in v0.45.0 and v0.8.0 respectively).
- Independent of v0.101.0 and v0.102.0 — can ship in parallel.

## Deliverables

### `rule_conflicts()` SQL Function

- [x] `pg_ripple.rule_conflicts(ruleset TEXT, mode TEXT DEFAULT 'static') → JSONB`
  - `mode = 'static'`: purely structural analysis of rule patterns — no VP table reads
  - `mode = 'runtime'`: scans the live triple store for already-derived contradictions
  - Returns a JSONB array of conflict objects (see schema below); empty array means no conflicts

### Static Analysis Mode

Detects two classes of structural contradiction at rule registration time or on demand:

- [x] **Same-head / conflicting-value conflicts**: pairs of rules with the same head predicate where variable patterns could produce semantically opposite values (e.g. one rule derives `?x ex:eligible true`, another derives `?x ex:eligible false` for the same subject)
- [x] **Rule-vs-SHACL conflicts**: pairs `(rule, SHACL constraint)` where the rule could derive `?x p ?v` but a `sh:not`/`sh:disjoint`/`sh:in` constraint forbids `?x p ?v` when the rule's conditions hold — fully decidable over the Datalog fragment pg_ripple supports (no negation-as-failure in head position, stratified negation only)
- [x] Detection implemented as pattern matching over the parsed rule AST and the compiled SHACL shape catalog; no SPI execution against VP tables
- [x] `pg_ripple.rule_conflict_check_on_load` GUC (BOOL, default `off`) — when `on`, static analysis runs automatically at `load_rules()` time and raises a WARNING for each conflict found (not an error — allows loading of rule sets that have known soft conflicts)
- [x] When v0.104.0's `install_rule_library()` is available, the same GUC gates conflict checking on library install — no additional integration code needed since `install_rule_library()` calls `load_rules()` internally

### Runtime Detection Mode

- [x] Queries `_pg_ripple.derivations` joined with VP tables to find:
  - Derived facts whose values violate active `sh:disjoint` or `sh:not` constraints
  - Derived facts where the same subject has two different values for a `sh:maxCount 1` property and both values were inferred (not just one explicit + one inferred)
- [x] `pg_ripple.block_on_conflict` GUC (BOOL, default `off`) — when `on`, the semi-naive inference engine calls `rule_conflicts(ruleset, 'runtime')` after each fixpoint iteration and raises PT0451 if any conflicts are found, halting inference before committing derived facts

### Conflict Report JSONB Schema

```json
[
  {
    "mode": "static" | "runtime",
    "rule_a": "<rule name>",
    "rule_b": "<rule name or null for SHACL>",
    "conflict_type": "same_head_opposing_values | rule_vs_shacl | runtime_violation",
    "head_predicate": "<IRI>",
    "conflicting_pattern": "<description>",
    "shacl_constraint": "<shape IRI or null>",
    "example_triple": {"s": "...", "p": "...", "o": "..."} | null
  }
]
```

### Error Catalog

- [x] PT0451: `inference halted: rule conflict detected in ruleset '%s' (set pg_ripple.block_on_conflict = off to continue despite conflicts)`

### REST Endpoint

- [x] `GET /rule-conflicts/{ruleset}?mode=static` and `GET /rule-conflicts/{ruleset}?mode=runtime` in `pg_ripple_http`

## Exit Criteria

- `cargo pgrx regress pg18` passes all tests including `v0103_conflicts_static` and `v0103_conflicts_runtime`

- `rule_conflicts()` in runtime mode finds existing derived contradictions in the test triple store
