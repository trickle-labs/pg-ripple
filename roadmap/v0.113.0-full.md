# v0.113.0 — A16 High: Bulk-load COPY Path & Performance Tuning (Full Details)

**Status: Released** | **Scope: Large**

## Overview

v0.113.0 addresses the remaining High-severity finding from Assessment 16 (H16-05:
bulk-load COPY path) and delivers four performance improvements (P4–P7) identified
in the same assessment cycle. The headline item is promoting `bulk_load_use_copy`
to default-on, yielding a 5–10× throughput gain for large triple loads. The release
also eliminates O(n) SPI round-trips in entity-resolution embedding, reduces HMAC
key-expansion overhead, and makes replication batch sizes and SSE buffer capacity
configurable via GUCs and environment variables.

> **Effort estimate: 2–3 person-weeks**

---

## Background: Why COPY Is Faster Than INSERT

PostgreSQL's `COPY ... FROM STDIN BINARY` bypasses the full query-planning and
expression-evaluation stack. For bulk VP table inserts:

- **INSERT path**: each row is planned individually, dictionary-encoded inline,
  and written one at a time — O(n) SPI calls, each with lock overhead.
- **COPY path**: a single staging table receives all encoded rows in one network
  transfer, then a single `INSERT … SELECT` join populates the VP table.

The COPY path was implemented in v0.94.0 as an opt-in feature for R2RML and CDC
bulk loads. This release promotes it to the default for `bulk_load_ntriples()` and
`bulk_load_turtle()`, closing the performance gap that existed since those functions
were introduced.

## Deliverables

### H16-05 — Bulk-load COPY Path

The `pg_ripple.bulk_load_use_copy` GUC (introduced in v0.94.0) is changed from
default `off` to default `on`:

```sql
-- Before (v0.112.0): GUC default = off
-- After (v0.113.0): GUC default = on
SET pg_ripple.bulk_load_use_copy = on;  -- now the default
```

Implementation changes in `src/gucs/storage.rs`:
- `BULK_LOAD_USE_COPY` constant default changed from `false` to `true`

Implementation changes in `src/gucs/registration/storage.rs`:
- GUC description updated to reflect default-on behaviour and throughput gain

The `bulk_load_ntriples()` and `bulk_load_turtle()` functions already call
`copy_into_vp()` when the GUC is on. No logic changes are required — only the
default flip.

Documentation update in `docs/src/cookbook/bulk-loading.md`:
- New "Performance" section with benchmark table showing 5–10× throughput gain
- Note that the COPY path requires a staging table and uses more transient disk space

### P4 — Batched Embedding HNSW Probe

**Problem:** `src/entity_resolution.rs` Stage 2 (`run_embedding_candidates`) called
`embedding_distance()` once per candidate pair — O(n) SPI round-trips per blocking
block, typically 10–200 calls per block.

**Fix:** Replace with a single batched `array_agg` CTE:

```sql
-- New batched probe (conceptual)
WITH candidates AS (
  SELECT unnest($1::text[]) AS iri
),
probes AS (
  SELECT c.iri, v.embedding
  FROM candidates c
  JOIN _pg_ripple.vector_index v ON v.subject_iri = c.iri
)
SELECT p.iri, p.embedding <=> $2::vector AS distance
FROM probes p
ORDER BY distance
LIMIT $3;
```

This reduces SPI round-trips from O(n) to O(1) per blocking block regardless of
block size.

### P5 — HMAC Clone Optimization

**Problem:** `src/pprl.rs::bloom_encode` re-keyed the HMAC instance on every
iteration of the `hash_count` loop, causing redundant key schedule expansion.

**Fix:** Create one base HMAC instance per encoding call and `clone()` it for each
hash position:

```rust
let base_hmac = Hmac::<Sha256>::new_from_slice(key).expect("valid key length");
for i in 0..hash_count {
    let mut h = base_hmac.clone();
    h.update(&[i as u8]);
    h.update(value.as_bytes());
    // ...
}
```

HMAC key schedule expansion is ~50ns on modern hardware. For 64 hash positions
(typical PPRL configuration) this saves ~3.2µs per `bloom_encode()` call.

### P6 — Replication Batch GUCs

**Problem:** `src/replication.rs` had hard-coded `const REPLICATION_BATCH_SIZE: i64 = 100`
and `const REPLICATION_BATCH_INTERVAL_MS: u64 = 500`. These could not be tuned
without recompiling.

**New GUCs:**

| GUC | Type | Default | Range | Description |
|-----|------|---------|-------|-------------|
| `pg_ripple.replication_batch_size` | `INT` | `100` | `1–10000` | Max events per replication batch |
| `pg_ripple.replication_batch_interval_ms` | `INT` | `500` | `10–60000` | Batch flush interval in milliseconds |

Both GUCs are defined in `src/gucs/storage.rs` and registered in
`src/gucs/registration/storage.rs`. The logical apply worker in `src/replication.rs`
reads them at runtime using `GucSetting::get()`.

### P7 — SSE Buffer Environment Variable

**Problem:** `pg_ripple_http/src/stream.rs` had `const SSE_CHANNEL_CAPACITY: usize = 256`
baked into the binary. High-throughput CDC deployments with many subscribers needed
a larger buffer; low-memory deployments wanted a smaller one.

**Fix:** Replace the constant with an environment-variable reader:

```rust
fn sse_channel_capacity() -> usize {
    std::env::var("PG_RIPPLE_HTTP_SSE_BUFFER")
        .ok()
        .and_then(|s| s.parse().ok())
        .unwrap_or(256)
        .clamp(1, 65536)
}
```

The value is read once at startup and cached. Valid range: 1–65536.
