## Feature 3 — `pg_ripple.compat_check()` SQL Function

### Background

The `COMPATIBLE_EXTENSION_MIN` constant in `pg_ripple_http` provides a startup
guard, but only from the HTTP companion's side. There is no SQL-level way to query
the compatibility status programmatically — a useful capability for monitoring
scripts and Kubernetes readiness probes that run SQL checks.

### Implementation

```sql
CREATE OR REPLACE FUNCTION pg_ripple.compat_check()
RETURNS jsonb
LANGUAGE sql STABLE
AS $$
  SELECT jsonb_build_object(
    'extension_version', extversion,
    'http_min_version', pg_ripple.http_min_version(),
    'compatible', extversion >= pg_ripple.http_min_version()
  )
  FROM pg_extension WHERE extname = 'pg_ripple';
$$;
```

The `http_min_version()` helper returns the `COMPATIBLE_EXTENSION_MIN` value
compiled into the currently-running `pg_ripple_http` binary via a GUC set at
HTTP companion startup.

### HTTP companion change

At startup, `pg_ripple_http` calls `SELECT pg_ripple.compat_check()` and refuses
to serve (`exit(1)`) if `compatible = false`, unless
`PG_RIPPLE_HTTP_SKIP_COMPAT_CHECK=1` is set.

## Feature 4 — Allen's Interval Relations

### Background

James F. Allen's 1983 paper "Maintaining Knowledge About Temporal Intervals"
defines 13 binary relations between time intervals (7 basic + 6 inverses). For
SPARQL FILTER and Datalog purposes, the 7 canonical relations are sufficient since
the inverses are expressed by swapping arguments.

### The Seven Relations

| Relation | Condition |
|----------|-----------|
| `pg:before(A, B)` | `A.end < B.start` |
| `pg:meets(A, B)` | `A.end = B.start` |
| `pg:overlaps(A, B)` | `A.start < B.start AND A.end > B.start AND A.end < B.end` |
| `pg:during(A, B)` | `A.start > B.start AND A.end < B.end` |
| `pg:finishes(A, B)` | `A.start > B.start AND A.end = B.end` |
| `pg:starts(A, B)` | `A.start = B.start AND A.end < B.end` |
| `pg:equals(A, B)` | `A.start = B.start AND A.end = B.end` |

### SPARQL FILTER Usage

```sparql
PREFIX pg: <http://pg-ripple.io/fn/>
SELECT ?event ?task WHERE {
  ?event pg_ripple:valid_from ?es ; pg_ripple:valid_to ?ee .
  ?task  pg_ripple:valid_from ?ts ; pg_ripple:valid_to ?te .
  FILTER(pg:overlaps(?es, ?ee, ?ts, ?te))
}
```

### Datalog Usage

```datalog
concurrent_events(?e1, ?e2) :-
  temporal_interval(?e1, ?s1, ?e1_end),
  temporal_interval(?e2, ?s2, ?e2_end),
  pg:overlaps(?s1, ?e1_end, ?s2, ?e2_end),
  ?e1 != ?e2.
```

### SQL Compilation

Each Allen's relation compiles to a `tsrange` expression:

```sql
-- pg:overlaps(a_start, a_end, b_start, b_end)
tsrange($1::timestamptz, $2::timestamptz) && tsrange($3::timestamptz, $4::timestamptz)
AND $1 < $3          -- A starts before B
AND $2 < $4          -- A ends before B ends
```

PostgreSQL's native `tsrange` overlap operator (`&&`) handles the base case;
additional conditions distinguish between `overlaps`, `during`, `starts`, etc.

## Feature 2 — Privacy Budget Registry

### Background

The differential privacy subsystem (v0.95.0) implements `dp_noisy_count()` and
`dp_noisy_histogram()` but has no budget accounting. Without budget tracking,
callers can query the same dataset indefinitely, accruing unbounded privacy loss.
The privacy budget registry enforces per-dataset, per-principal epsilon budgets.

### Schema

```sql
CREATE TABLE _pg_ripple.privacy_budget (
  dataset_id    BIGINT      NOT NULL,
  principal     TEXT        NOT NULL,
  budget_total  FLOAT8      NOT NULL CHECK (budget_total > 0),
  budget_spent  FLOAT8      NOT NULL DEFAULT 0 CHECK (budget_spent >= 0),
  last_reset_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (dataset_id, principal)
);
```

### Budget Enforcement

Before executing a DP query, the system checks:
```sql
SELECT budget_total - budget_spent >= $epsilon
FROM _pg_ripple.privacy_budget
WHERE dataset_id = $dataset_id AND principal = $principal;
```

If false → raise `PT0490: privacy budget exhausted for dataset/principal`.

After a successful query:
```sql
UPDATE _pg_ripple.privacy_budget
SET budget_spent = budget_spent + $epsilon
WHERE dataset_id = $dataset_id AND principal = $principal;
```

### Automatic Reset

The merge background worker resets budgets that have exceeded the reset interval:
```sql
UPDATE _pg_ripple.privacy_budget
SET budget_spent = 0, last_reset_at = now()
WHERE last_reset_at < now() - current_setting('pg_ripple.privacy_budget_reset_interval')::interval;
```

GUC: `pg_ripple.privacy_budget_reset_interval TEXT DEFAULT '1 day'`

### HTTP Endpoint

```
GET /dp/budget/{dataset}/{principal}
→ { "dataset_id": 123, "principal": "alice", "budget_total": 1.0, "budget_spent": 0.35, "remaining": 0.65 }
```

## Feature 1 — Integrated Benchmark Runner

### Schema

```sql
CREATE TABLE _pg_ripple.bench_history (
  run_id              BIGSERIAL PRIMARY KEY,
  profile             TEXT NOT NULL,
  started_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
  duration_ms         BIGINT,
  triples_processed   BIGINT,
  queries_per_second  FLOAT8
);
```

### SQL API

```sql
-- Run BSBM profile and return run_id
SELECT pg_ripple.bench_workload('bsbm');

-- View history
SELECT * FROM _pg_ripple.bench_history ORDER BY started_at DESC LIMIT 10;
```

### Supported Profiles

| Profile | Benchmark | What it measures |
|---------|-----------|-----------------|
| `bsbm` | Berlin SPARQL Benchmark | Multi-query throughput |
| `watdiv` | WatDiv | Query shape coverage |
| `pagerank` | Custom | PageRank convergence time |
| `pprl` | Custom | PPRL encode throughput |

### HTTP Endpoint

```
GET /admin/bench-history?limit=20&profile=bsbm
→ [{ "run_id": 1, "profile": "bsbm", "started_at": "...", "duration_ms": 4521, ... }]
```
