### L15-06: `gen_random_uuid()` availability check at `_PG_init`

In `_PG_init`, after the extension initialises, attempt:
```sql
SELECT gen_random_uuid()
```
If it raises `ERROR:  function gen_random_uuid() does not exist`, emit a
`WARNING` level PostgreSQL log message directing the user to:
```
HINT: run CREATE EXTENSION IF NOT EXISTS pgcrypto; to enable gen_random_uuid()
```
This guards against installations where `pgcrypto` is not pre-loaded, which
would cause the bidi relay and SPARQL UUID functions to fail at runtime.

**Note**: In PostgreSQL 14+ `gen_random_uuid()` is a built-in and does not
require pgcrypto. The check is a safety guard for unusual build configurations.

### L15-11: Document `statement_id_seq` exhaustion behaviour

Add a new "Sequence Exhaustion" section to `docs/src/operations/scaling.md`
describing:

- `statement_id_seq` is a `BIGINT` sequence (max ~9.2 × 10^18)
- At 1M inserts/second, exhaustion takes ~292,000 years
- At 1B inserts/second (theoretical max), exhaustion takes ~292 years
- Behaviour on exhaustion: PostgreSQL raises `ERROR: nextval: reached maximum value of sequence`
- Mitigation: `pg_ripple.sid_exhaustion_years()` returns years remaining
- Recovery: `ALTER SEQUENCE _pg_ripple.statement_id_seq RESTART WITH 1` (requires
  clearing all VP tables first — document the full procedure)
