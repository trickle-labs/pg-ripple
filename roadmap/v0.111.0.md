# v0.111.0 — Privacy-Preserving Record Linkage (PPRL) Primitives

**Theme**: Bloom-filter PPRL and differential-privacy aggregates for cross-organization entity resolution.

> **In plain language:** When two organizations want to find records that refer to the same
> real-world person — patients across hospital systems, customers across banks — they typically
> cannot share the raw data (names, dates of birth, SSNs) due to privacy regulations.
> This release adds Bloom-filter encoding that converts sensitive identifiers into
> fixed-length bit vectors that can be safely exchanged: the bit vectors support similarity
> comparison without exposing the underlying values. Combined with pg_ripple's SPARQL
> federation (`SERVICE`), two organizations can find their shared patients without either
> side ever seeing the other's raw PII.
>
> **Foundation document**: [plans/neuro-symbolic-record-linkage.md](../plans/neuro-symbolic-record-linkage.md)
> §11.3 (PPRL worked example), §15.1 (gap: Bloom-filter primitives), §13.5 (threat model).
>
> **Reference**: Schnell, Bachteler & Reiher (2009) — "Privacy-preserving record linkage
> using Bloom filters." *BMC Medical Informatics and Decision Making* 9:41.
>
> **Effort estimate: 2–3 person-weeks**

## Deliverables

### Bloom-Filter Encoding

- [ ] `pg_ripple.bloom_encode(value TEXT, key TEXT, hash_count INT DEFAULT 30, length INT DEFAULT 1024) → BIT VARYING`
  - Implements the CLK (Cryptographic Longterm Key) construction: for each of `hash_count` independent SHA-256 HMACs (keyed with `key || i` for i in 0..hash_count), maps the hash to a bit position in a `length`-bit vector and sets that bit
  - `pgcrypto`'s `hmac(data, key, 'sha256')` is used for the HMAC — no custom C code, fully safe Rust via `pgrx::SpiClient`
  - `value` length validated against `pg_ripple.bloom_max_input_length` GUC (INT, default: 4096 bytes) — raises PT0470 on violation
  - `hash_count` must be in [1, 256], `length` must be in [64, 65536] — raises PT0471 on invalid parameters
  - **Security documentation**: values of `hash_count < 30` or `length < 1024` significantly increase the risk of re-identification (graph-based attack); these defaults are chosen to match the recommended parameters from Schnell et al. A warning is emitted to the PostgreSQL log when parameters are below the recommended minimums
- [ ] PT0470 entry in error catalog (`bloom_encode: input length %d exceeds bloom_max_input_length GUC (%d)`)
- [ ] PT0471 entry in error catalog (`bloom_encode: hash_count %d or length %d outside valid range`)

### Dice Similarity SPARQL Function and Datalog Builtin

- [ ] `pg:dice_similarity(?a, ?b) → xsd:double` SPARQL FILTER function — computes Dice coefficient: `2 * popcount(a & b) / (popcount(a) + popcount(b))`; returns 0.0 if both bit strings are zero; raises type error if arguments are not `BIT VARYING` literals
- [ ] `pg:dice_similarity(?a, ?b)` as a Datalog built-in predicate for use in rule bodies
- [ ] Matching SQL function `pg_ripple.dice_similarity(a BIT VARYING, b BIT VARYING) → FLOAT8` usable outside SPARQL
- [ ] pg_regress test: `dice_similarity(bloom_encode('Alice', 'k', 30, 1024), bloom_encode('Alice', 'k', 30, 1024)) = 1.0`; `dice_similarity(bloom_encode('Alice', 'k'), bloom_encode('Bob', 'k')) < 0.5`

### Differential-Privacy Aggregate Functions

- [ ] `pg_ripple.dp_noisy_count(query TEXT, epsilon FLOAT8) → BIGINT`
  - **SQL injection mitigation**: `query` must begin with `SELECT` (case-insensitive, whitespace-trimmed); the function rejects any query containing `;`, `INSERT`, `UPDATE`, `DELETE`, `DROP`, `CREATE`, `ALTER`, `GRANT`, `REVOKE`, or `TRUNCATE` keywords (case-insensitive) and raises PT0474 if validation fails
  - Executes `query` via SPI in read-only mode (`SPI_execute_with_args` with `read_only = true`), expects a single-row single-column result, adds Laplace noise $\text{Lap}(0, 1/\epsilon)$ to the count
  - `epsilon` validated to `(0.0, 10.0]` — raises PT0472 on out-of-range values
  - Query must return a single BIGINT — raises PT0473 if result is not a single BIGINT
  - Result always ≥ 0 (negative noisy counts are clamped to 0)
- [ ] `pg_ripple.dp_noisy_histogram(query TEXT, key_column TEXT, count_column TEXT, epsilon FLOAT8) → TABLE(key TEXT, noisy_count BIGINT)`
  - Same SQL injection validation as `dp_noisy_count` (read-only SELECT only, raises PT0474 on rejection)
  - Executes `query` in read-only mode, reads `key_column` and `count_column`, adds independent Laplace noise to each bucket
  - Same `epsilon` validation as `dp_noisy_count`
- [ ] PT0472 entry in error catalog (`dp_noisy_count: epsilon %g out of valid range (0, 10]`)
- [ ] PT0473 entry in error catalog (`dp_noisy_count: query must return a single INTEGER value`)
- [ ] PT0474 entry in error catalog (`dp_noisy_count: query rejected by validation — must be a read-only SELECT`)
- [ ] pg_regress test: `dp_noisy_count('SELECT COUNT(*) FROM _pg_ripple.dictionary', 0.1)` returns a non-negative integer (exact value non-deterministic; test asserts `>= 0 AND NOT NULL`)

### Documentation: PPRL Cookbook

- [ ] `docs/src/cookbook/pprl.md` — end-to-end worked example:
  1. Each organization encodes patient identifiers as Bloom-filter triples using `bloom_encode()` and stores them in a named graph
  2. The coordinating party (or each party) runs a SPARQL query with `SERVICE` federation: the remote endpoint exposes only `pprl:nameBloom` triples, not raw PII
  3. `FILTER(pg:dice_similarity(?bloom_a, ?bloom_b) > 0.85)` surfaces candidate matches above the threshold
  4. Optional: `dp_noisy_count()` for aggregate-only reporting (e.g. "approximately N shared patients") without exposing individual identities
  - Chapter includes security notes: recommended CLK parameters, risks of low hash-count/short vectors, key management (shared key must be established out-of-band via a secure channel), and a brief note on the patent status of CLK (none — published academic work)
  - Links to the Schnell et al. reference
