# v0.94.0 Full Details — Assessment 15 High Remediation

## Overview

v0.94.0 closes four high-severity findings from Assessment 15 of the pg_ripple
production readiness review:

| ID | Severity | Finding | Fix |
|----|----------|---------|-----|
| H15-01 | High | `bump-version` auto-sets `COMPATIBLE_EXTENSION_MIN` to `NEW_VERSION` unconditionally | Accept optional `COMPAT_MIN` param; fix `check-version-sync` to allow `COMPAT_MIN ≤ EXT_VER` |
| H15-02 | High | `ddl_guard_vp_tables()` SECURITY DEFINER has no pinned `search_path` | Add `SET search_path = pg_catalog, _pg_ripple, public`; add CI lint |
| H15-03/L15-13 | High/Low | bidi relay has no inflight limit; can OOM under load | `bidi_relay_max_inflight` GUC + drop-oldest gate + Prometheus counter |
| H15-05/M15-20 | High/Medium | Bulk loader inlines VP insert; no shared helper; no COPY path | Extract `copy_into_vp()` helper + `bulk_load_use_copy` GUC |

## Detailed Changes

### H15-01: `just bump-version` improvements

The previous `bump-version` recipe always attempted to set `COMPATIBLE_EXTENSION_MIN`
to `NEW_VERSION` (replacing `OLD_VERSION`). Since `COMPATIBLE_EXTENSION_MIN` is
intended to be a *minimum* (trailing behind the current version by at least one
release), this conflated "version bump" with "compat min bump".

**Changes:**
- `bump-version NEW_VERSION [COMPAT_MIN]` — optional second argument sets
  `COMPATIBLE_EXTENSION_MIN` explicitly; when omitted, it is left unchanged.
- `pg_ripple.control` `comment` line is now updated atomically.
- `check-version-sync` recipe updated: checks `COMPAT_MIN ≤ EXT_VER` via
  `sort -V`, not `COMPAT_MIN == EXT_VER`. This matches the CI `lint-version-sync`
  gate in `.github/workflows/ci.yml`.

**For v0.94.0:**
- `COMPATIBLE_EXTENSION_MIN` set to `"0.93.0"` (pg_ripple_http requires pg_ripple ≥ 0.93.0).

### H15-02: SECURITY DEFINER search_path hardening

PostgreSQL SECURITY DEFINER functions run with the privileges of the definer, but
the caller can manipulate `search_path` to redirect unqualified name lookups into
attacker-controlled schemas. The fix is to pin `search_path` in the function
definition using `SET search_path = …`.

**Changed function:**
```sql
CREATE OR REPLACE FUNCTION _pg_ripple.ddl_guard_vp_tables()
    RETURNS event_trigger
    LANGUAGE plpgsql
    SECURITY DEFINER
    SET search_path = pg_catalog, _pg_ripple, public
AS $$ … $$;
```

**New CI script:** `scripts/check_security_definer_search_path.sh`
- Scans `src/**/*.rs` for SECURITY DEFINER occurrences (skipping comment lines).
- Fails if any non-comment SECURITY DEFINER line lacks a `SET search_path` clause
  within ±8 lines.
- Wired into `ci.yml` as `SECURITY DEFINER search_path lint` step.

### H15-03/L15-13: Bounded bidi relay channel

**Problem:** The `ingest_jsonld_impl()` function in `src/bidi/relay.rs` dispatches
relay events synchronously without any concurrency limit. Under load (e.g., a
CDC flood), multiple sessions can simultaneously hold SPI connections inside
relay dispatch, potentially exhausting backend memory.

**Implementation:**

Static counters in `src/stats.rs`:
```rust
pub(crate) static BIDI_RELAY_INFLIGHT: AtomicI64 = AtomicI64::new(0);
pub(crate) static BIDI_RELAY_DROPPED_TOTAL: AtomicI64 = AtomicI64::new(0);
```

Helper functions:
- `relay_inflight_acquire() -> bool` — CAS check against `bidi_relay_max_inflight`;
  increments inflight on success; increments dropped on failure.
- `relay_inflight_release()` — decrements inflight counter.

GUC: `pg_ripple.bidi_relay_max_inflight` (default: 1000, range: 1–100000, Suset).

`ingest_jsonld_impl()` guards:
```rust
if !crate::stats::relay_inflight_acquire() {
    pgrx::warning!("bidi relay inflight limit reached; dropping ingest_jsonld call");
    return 0;
}
// … do work …
crate::stats::relay_inflight_release();
```

`streaming_metrics()` returns `bidi_relay_inflight` and `bidi_relay_dropped_total`.

Prometheus: `pg_ripple_bidi_relay_dropped_total` counter in `pg_ripple_http/src/metrics.rs`
and `/metrics` endpoint.

### H15-05/M15-20: COPY-style bulk load path

**Problem:** `batch_insert_encoded()` in `src/storage/ops/mod.rs` formats integer
triples into SQL VALUES strings, allocating one `String` per row. For large loads
(100k+ triples), this allocates significant heap memory and generates large SQL
statement strings. A COPY-style path using UNNEST arrays avoids per-row allocation.

**New helper in `src/storage/ops/mod.rs`:**
```rust
pub(crate) fn copy_into_vp(delta: &str, rows: &[(i64, i64, i64)]) {
    // Collect three BIGINT[] arrays and INSERT via UNNEST.
    let sql = format!(
        "INSERT INTO {delta} (s, o, g)
         SELECT s, o, g FROM UNNEST($1::bigint[], $2::bigint[], $3::bigint[]) AS t(s, o, g)
         ON CONFLICT (s, o, g) DO NOTHING"
    );
    // … Spi::run_with_args with array parameters …
}
```

`batch_insert_encoded()` dispatches to `copy_into_vp()` when `bulk_load_use_copy = on`.
The helper is `pub(crate)` so R2RML and CDC can call it directly.

**GUC:** `pg_ripple.bulk_load_use_copy` (default: off, Userset).

## Migration Notes

The only SQL-visible change is the recreation of `_pg_ripple.ddl_guard_vp_tables()`
with the `SET search_path` clause. The migration script
`sql/pg_ripple--0.93.0--0.94.0.sql` applies this `CREATE OR REPLACE FUNCTION`
statement and records the schema version.

New GUCs (`bidi_relay_max_inflight`, `bulk_load_use_copy`) are registered at
Postmaster startup and require no SQL migration.
