# v0.108.0 — Bayesian Confidence Updates

**Theme**: Dynamic belief revision — update confidence scores as new evidence arrives.

> **In plain language:** v0.87.0 added confidence scores to facts: "we believe Alice works at
> Acme Corp with 85% confidence." But that 85% was static — it never changed. In the real
> world, evidence accumulates. You receive a new document confirming Alice's employment. A
> second source contradicts it. An auditor marks the original source as unreliable. Each piece
> of evidence should *update* the confidence, not replace it wholesale. This release implements
> Bayesian belief revision: as new evidence arrives, the confidence score is updated using
> Bayes' theorem, and that update propagates forward through any facts that were *derived from*
> the updated fact. The knowledge graph becomes a living, self-updating belief system.
>
> **Foundation document**: [plans/expert-system.md](../plans/expert-system.md)
>
> **Effort estimate: 3–4 person-weeks**

## Prerequisites

- v0.107.0 released and all tests passing.
- v0.87.0 probabilistic Datalog layer available (`@weight` annotations, noisy-OR propagation).

## Deliverables

### `update_confidence()` SQL Function

- [ ] `pg_ripple.update_confidence(subject TEXT, predicate TEXT, object TEXT, evidence JSONB) → FLOAT8`
  - `evidence` format: `{"source": "...", "likelihood_ratio": 2.5}` — the likelihood ratio $\Lambda = P(\text{evidence} \mid \text{fact true}) / P(\text{evidence} \mid \text{fact false})$
  - Retrieves the current prior confidence $p_0$ for the fact via the dictionary + RDF-star confidence annotation
  - Computes the posterior: $p_1 = \frac{\Lambda \cdot p_0}{\Lambda \cdot p_0 + (1 - p_0)}$ (Bayes' theorem in odds form)
  - Clamps result to `[0.001, 0.999]` (never absolute certainty — documented)
  - Updates the `ex:confidence` RDF-star annotation in-place
  - Returns the new confidence value
  - `likelihood_ratio` must be positive — raises PT0440 if ≤ 0.0
  - `pg_ripple.confidence_update_strategy` GUC (`'bayesian'` | `'noisy-or'` | `'manual'`, default `'bayesian'`):
    - `'bayesian'`: applies the formula above
    - `'noisy-or'`: delegates to the existing v0.87 noisy-OR combiner with the evidence weight as a new input
    - `'manual'`: `update_confidence()` raises PT0441 (caller must set confidence directly via `insert_triple`)

### Evidence Log

- [ ] `_pg_ripple.evidence_log (id BIGSERIAL PRIMARY KEY, sid BIGINT NOT NULL REFERENCES _pg_ripple.dictionary(id), event_at TIMESTAMPTZ DEFAULT now(), source_iri BIGINT, likelihood_ratio FLOAT8 NOT NULL, prior_confidence FLOAT8 NOT NULL, posterior_confidence FLOAT8 NOT NULL)` — append-only log of every evidence event
- [ ] `pg_ripple.evidence_log_retention` GUC (INTERVAL, default `'1 year'`) — rows older than this are pruned by a background worker pass
- [ ] `pg_ripple.vacuum_evidence_log() → BIGINT` — prunes expired rows and returns count removed

### Downstream Confidence Propagation via DRed

- [ ] When `update_confidence()` changes a base-fact confidence, downstream inferred-fact confidences update incrementally:
  - Identifies all derived facts whose derivation chain includes the updated base fact (via `_pg_ripple.derivations`)
  - Re-propagates confidence through the derivation DAG using the current `confidence_update_strategy`
- [ ] `pg_ripple.confidence_propagation_max_depth` GUC (INT, default: 10) — caps cascade depth; facts beyond max depth are recorded in `_pg_ripple.confidence_stale (sid BIGINT PRIMARY KEY, marked_at TIMESTAMPTZ)` and reprocessed by the background worker
- [ ] **Background worker**: a new `confidence_reprocessing` mode added to the existing merge background worker; fires on a schedule controlled by `pg_ripple.confidence_reprocessing_interval` GUC (INTERVAL, default `'30 seconds'`); drains `_pg_ripple.confidence_stale` and re-propagates

### Conflict-Weighted Confidence

- [ ] When `rule_conflicts()` (v0.103.0) detects that two rules derive contradictory conclusions, attenuate both derived-fact confidence scores:
  - Attenuation factor = `1.0 - (conflict_severity * pg_ripple.conflict_confidence_penalty)` where `conflict_severity` is `1.0` for full `sh:disjoint` conflicts and `0.5` for same-head opposing-value conflicts
  - `pg_ripple.conflict_confidence_penalty` GUC (FLOAT8 in [0.0, 1.0], default: `0.3`)

### Bulk Evidence Ingestion

- [ ] `pg_ripple.bulk_update_confidence(data TEXT, format TEXT DEFAULT 'csv') → BIGINT`
  - Accepts CSV or JSON-L data: each row specifies `subject, predicate, object, source, likelihood_ratio`
  - Processes rows in batches (batch size controlled by `pg_ripple.confidence_batch_size` GUC, default: 1000)
  - Returns count of facts updated
  - Idempotent within a transaction: duplicate rows for the same fact are collapsed into a single update (product of likelihood ratios)

### REST Endpoint

- [ ] `POST /confidence/update` in `pg_ripple_http` — request body: `{"subject":"...","predicate":"...","object":"...","evidence":{"source":"...","likelihood_ratio":2.5}}`, response: `{"prior": 0.75, "posterior": 0.86}`
- [ ] `POST /confidence/bulk-update` — request body: CSV or JSON-L; response: `{"updated": 1234}`

### Proptest Suite

- [ ] `tests/proptest/bayesian_confidence.rs`:
  - **Monotone property**: a likelihood ratio > 1.0 always increases confidence; < 1.0 always decreases it
  - **Bayesian consistency**: applying a sequence of evidence updates then computing the joint posterior equals applying all evidence jointly (sequential Bayesian updates commute when evidence is conditionally independent)
  - **Propagation property**: updating a base fact's confidence changes all derived facts that depend on it, and none that do not
  - All properties tested against a pure-Rust reference oracle implementing the same Bayesian formula

## Exit Criteria

- `cargo pgrx regress pg18` passes all tests including `v0108_confidence`
- `update_confidence()` with `likelihood_ratio > 1.0` increases the returned confidence value
- `update_confidence()` with `likelihood_ratio < 1.0` decreases it
- Each call creates one row in `_pg_ripple.evidence_log`
- Updating a base-fact confidence causes downstream derived-fact confidences to change
- Facts beyond `confidence_propagation_max_depth` appear in `_pg_ripple.confidence_stale`
- PT0440 is raised for non-positive likelihood ratios
- All proptest properties pass
- `POST /confidence/update` REST endpoint returns a JSON object with `prior` and `posterior`
