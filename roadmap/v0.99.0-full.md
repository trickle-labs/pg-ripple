# v0.99.0 — DCTERMS, Schema.org & FOAF Vocabulary Bundles (Full Details)

**Status: Planned** | **Scope: Medium**

## Overview

v0.99.0 completes the "Big 5" foundational vocabulary stack for pg_ripple by delivering native Datalog rule sets and SHACL shape bundles for DCTERMS, Schema.org, and FOAF — the three most widely-used vocabularies in linked data that were not yet covered. All three are delivered as named bundles via the `load_datalog_bundle()` / `load_shape_bundle()` API introduced in v0.98.0, and each includes cross-vocabulary bridge rules that unlock combined reasoning.

## Background

After SKOS (v0.98.0), a survey of ontologist usage patterns shows that the next three highest-frequency vocabularies in real-world RDF datasets are:

| Vocabulary | Namespaces | Typical use |
|---|---|---|
| Dublin Core Terms | `dcterms:`, `dc11:` | Metadata on every resource: creator, date, subject, rights, format |
| Schema.org | `schema:` | Entity typing: Person, Organization, Product, Place, Event |
| FOAF | `foaf:` | Social graph: Person, knows, mbox, homepage, account |

These three together with SKOS and OWL 2 RL (already in pg_ripple) cover the metadata and entity layers of ~90% of public linked-data datasets (DBpedia, Wikidata, data.gov, schema.org markup, library catalogs).

## DCTERMS (`"dcterms"` bundle)

### Vocabulary scope

Dublin Core Terms is the universally-adopted metadata vocabulary. It supersedes the older `dc:` / `dc11:` namespace. Key property groups:

- **Descriptive**: `dcterms:title`, `dcterms:description`, `dcterms:subject`, `dcterms:language`, `dcterms:format`
- **Attribution**: `dcterms:creator`, `dcterms:contributor`, `dcterms:publisher`, `dcterms:rights`, `dcterms:license`
- **Temporal**: `dcterms:created`, `dcterms:modified`, `dcterms:dateAccepted`, `dcterms:dateSubmitted`
- **Structural**: `dcterms:hasPart`/`dcterms:isPartOf`, `dcterms:hasVersion`/`dcterms:isVersionOf`, `dcterms:replaces`/`dcterms:isReplacedBy`

### Datalog rules (11 rules, bundle `"dcterms"`)

```datalog
# DC11 backwards compatibility: dc:subject → dcterms:subject, dc:creator → dcterms:creator, etc.
dcterms:subject(?s, ?o)      :- dc11:subject(?s, ?o).
dcterms:creator(?s, ?o)      :- dc11:creator(?s, ?o).
dcterms:title(?s, ?o)        :- dc11:title(?s, ?o).
dcterms:description(?s, ?o)  :- dc11:description(?s, ?o).
dcterms:date(?s, ?o)         :- dc11:date(?s, ?o).

# Inverse structural relations
dcterms:isPartOf(?part, ?whole) :- dcterms:hasPart(?whole, ?part).
dcterms:hasPart(?whole, ?part)  :- dcterms:isPartOf(?part, ?whole).
dcterms:isVersionOf(?v, ?base)  :- dcterms:hasVersion(?base, ?v).
dcterms:isReplacedBy(?old, ?new):- dcterms:replaces(?new, ?old).

# dcterms:subject → skos:Concept type assertion (when SKOS bundle active)
rdf:type(?subject, skos:Concept) :- dcterms:subject(?doc, ?subject),
                                    rdf:type(?subject, skos:Concept).  # DC-SKOS-01
```

### SHACL shapes (8 validators, bundle `"dcterms-integrity"`)

| Constraint | W3C/ISO basis | What it checks |
|---|---|---|
| `dcterms:title` MinCount 1 | — | Every described resource should have a title |
| `dcterms:created` datatype xsd:date/xsd:dateTime | — | Date literals must be valid XSD date types |
| `dcterms:language` pattern BCP-47 | RFC 5646 | Language tags like `"en"`, `"en-GB"` |
| `dcterms:creator` nodeKind IRI/BlankNode | — | Creator must be a resource, not a plain literal |
| `dcterms:license` nodeKind IRI | — | License must be a dereferenceable IRI |
| `dcterms:modified` ≥ `dcterms:created` | — | Modification date cannot precede creation |
| `dcterms:format` vs `dcterms:MediaType` | — | Format values should be media-type IRIs |
| `dcterms:hasPart` no self-reference | — | A resource cannot be its own part |

## Schema.org (`"schema"` bundle)

### Vocabulary scope

Schema.org defines ~800 types and ~1,400 properties for structured web data. The Datalog bundle targets the 15 most inferentially useful rules across the type hierarchy and inverse-property pairs. Full Schema.org type-hierarchy reasoning (all 800+ types) is handled by loading the Schema.org RDFS description into the graph and activating OWL RL rules; the bundle provides the highest-value shortcuts.

### Datalog rules (15 rules, bundle `"schema"`)

```datalog
# Inverse property pairs
schema:isPartOf(?part, ?whole)        :- schema:hasPart(?whole, ?part).
schema:hasPart(?whole, ?part)         :- schema:isPartOf(?part, ?whole).
schema:memberOf(?person, ?org)        :- schema:member(?org, ?person).
schema:subjectOf(?thing, ?creative)   :- schema:about(?creative, ?thing).

# Type hierarchy shortcuts (most common inheritance chains)
rdf:type(?x, schema:Thing)            :- rdf:type(?x, schema:Person).
rdf:type(?x, schema:Thing)            :- rdf:type(?x, schema:Organization).
rdf:type(?x, schema:Thing)            :- rdf:type(?x, schema:Product).
rdf:type(?x, schema:Organization)     :- rdf:type(?x, schema:LocalBusiness).
rdf:type(?x, schema:Organization)     :- rdf:type(?x, schema:EducationalOrganization).
rdf:type(?x, schema:CreativeWork)     :- rdf:type(?x, schema:Article).
rdf:type(?x, schema:CreativeWork)     :- rdf:type(?x, schema:Dataset).
rdf:type(?x, schema:Event)            :- rdf:type(?x, schema:SocialEvent).

# Cross-vocabulary bridges
foaf:maker(?thing, ?person)    :- schema:author(?thing, ?person).   # SCHEMA-FOAF-01
dcterms:creator(?thing, ?p)    :- schema:author(?thing, ?p).         # SCHEMA-DC-01
rdf:type(?x, schema:Dataset)  :- rdf:type(?x, dcat:Dataset).        # SCHEMA-DCAT-01
```

### SHACL shapes (6 validators, bundle `"schema-integrity"`)

| Constraint | What it checks |
|---|---|
| `schema:name` MinCount 1 on `schema:Thing` | All schema-typed resources should be named |
| `schema:url` nodeKind IRI | URL properties must be IRIs, not plain strings |
| `schema:price` datatype xsd:decimal | Prices must be numeric |
| `schema:startDate` ≤ `schema:endDate` | Event date ordering |
| `schema:email` pattern `^[^@]+@[^@]+$` | Basic email format |
| `schema:telephone` datatype xsd:string, pattern E.164 | Phone number format |

### SQL helper

```sql
-- Returns all Schema.org type ancestors for an IRI (via type hierarchy closure)
SELECT * FROM pg_ripple.schema_type_ancestors('https://schema.org/LocalBusiness');
-- → schema:Organization, schema:Thing
```

## FOAF (`"foaf"` bundle)

### Vocabulary scope

FOAF describes people, their activities, and their relations to other people and objects. Core properties: `foaf:name`, `foaf:knows`, `foaf:mbox`, `foaf:homepage`, `foaf:account`, `foaf:made`/`foaf:maker`, `foaf:Organization`, `foaf:Person`, `foaf:Agent`.

### Datalog rules (8 rules, bundle `"foaf"`)

```datalog
# Symmetric relations
foaf:knows(?b, ?a)             :- foaf:knows(?a, ?b).

# Type subsumption
rdf:type(?x, foaf:Agent)       :- rdf:type(?x, foaf:Person).
rdf:type(?x, foaf:Agent)       :- rdf:type(?x, foaf:Organization).
rdf:type(?x, foaf:Agent)       :- rdf:type(?x, foaf:Group).

# Inverse properties
foaf:accountFor(?acct, ?agent) :- foaf:account(?agent, ?acct).
foaf:made(?agent, ?thing)      :- foaf:maker(?thing, ?agent).
foaf:maker(?thing, ?agent)     :- foaf:made(?agent, ?thing).

# DC bridge: dcterms:creator → foaf:maker
foaf:maker(?thing, ?agent)     :- dcterms:creator(?thing, ?agent).  # DC-FOAF-01
```

### SHACL shapes (5 validators, bundle `"foaf-integrity"`)

| Constraint | What it checks |
|---|---|
| `foaf:homepage` nodeKind IRI | Homepage must be a dereferenceable IRI |
| `foaf:mbox` pattern `^mailto:` | Mailbox values must use the `mailto:` scheme |
| `foaf:name` datatype xsd:string | Names must be plain string literals |
| `foaf:mbox_sha1sum` pattern `^[0-9a-f]{40}$` | SHA1 hash format |
| `foaf:Person` MaxCount 1 for `foaf:mbox` | Unique mailbox per person |

### SQL helper

```sql
-- Returns all foaf:Person IRIs and their foaf:name labels
SELECT * FROM pg_ripple.foaf_persons();
-- → (person_iri, name_label)
```

## Cross-vocabulary bridge rules

v0.99.0 adds three cross-vocabulary bridge rules that activate when multiple bundles are loaded simultaneously:

| Rule ID | Description | Bundles required |
|---|---|---|
| DC-FOAF-01 | `dcterms:creator` → `foaf:maker` inference | `dcterms` + `foaf` |
| DC-SKOS-01 | `dcterms:subject` nodes asserted as `skos:Concept` when SKOS scheme present | `dcterms` + `skos` |
| SCHEMA-FOAF-01 | `schema:author` → `foaf:maker` bridge | `schema` + `foaf` |
| SCHEMA-DC-01 | `schema:author` → `dcterms:creator` bridge | `schema` + `dcterms` |
| SCHEMA-DCAT-01 | `dcat:Dataset` → `schema:Dataset` type assertion | `schema` (DCAT prefix only) |

Cross-bundle rules activate automatically via implicit dependency resolution in `load_datalog_bundle()` when the required co-bundles are both active.
