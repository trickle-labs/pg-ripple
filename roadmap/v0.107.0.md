# v0.107.0 — Temporal Reasoning Phase 2: Sequential Patterns & CDC Integration

**Theme**: Detect sequences and patterns in time-ordered facts; wire temporal recording to the CDC replication log.

> **In plain language:** v0.106.0 let you ask "which facts were valid at a given time?" This
> release asks harder questions: "did event A happen before event B within the last 30 days?"
> or "did a patient's fever reading exceed 38°C on three consecutive days?" These sequential
> patterns are essential for clinical alerting, fraud detection, and equipment monitoring. At
> the same time, this release integrates the temporal layer with CDC (change data capture) so
> that when a new fact is recorded in the database, its validity timestamp is set automatically
> from the transaction clock — no manual timestamp management required.
>
> **Foundation document**: [plans/expert-system.md](../plans/expert-system.md)
>
> **Effort estimate: 4–5 person-weeks**

## Prerequisites

- v0.106.0 released and all tests passing (`_pg_ripple.temporal_facts` table available, `AFTER`/`BEFORE`/`DURING` operators working).
- CDC bridge infrastructure available (since v0.52.0) — the integration modifies the CDC assertion/retraction path in `src/storage/cdc_bridge.rs` to route temporal-predicate writes through `temporal_facts` instead of VP tables.

## Deliverables

### Three New Temporal Operators

- [x] `WITHIN(?s, ?p, ?o, duration)` — true if `?s ?p ?o` holds at least once within the most recent `duration` interval relative to query time; example: `WITHIN(?patient, ex:fever, ?temp, 'P3D')` matches patients with a fever reading in the last 3 days
- [x] `SEQUENCE(event1, event2, window)` — true if `event1` (a `(s, p, o)` triple pattern) occurs strictly before `event2` within a sliding `window` duration; "occurs" means `valid_from` of event1 < `valid_from` of event2 and both fall within `window` of each other; example: `SEQUENCE(?x ex:loginAttempt ?failed, ?x ex:accountLocked ?t, 'PT1H')` detects lockouts within 1 hour of failed logins
- [x] `CONSECUTIVE(n, predicate, window)` — true if there exist `n` rows for the same subject and given predicate in `temporal_facts` where each successive `valid_from` is strictly greater than the previous and all `n` fall within `window` duration of the first; example: `CONSECUTIVE(3, ex:feverReading, 'P3D')` detects three fever readings on consecutive days

### SQL Compilation of Sequential Operators

All three operators compile to standard PostgreSQL window functions over `temporal_facts`:

- [x] `WITHIN` → `EXISTS (SELECT 1 FROM temporal_facts WHERE s = ... AND p = ... AND valid_from >= now() - interval)` with index-only scan on the partial index
- [x] `SEQUENCE` → correlated subquery using `LAG`/`LEAD` window functions partitioned by `(s, p)` ordered by `valid_from`; the query optimizer can use the `(s, p, valid_from)` B-tree index
- [x] `CONSECUTIVE` → `ROW_NUMBER() OVER (PARTITION BY s, p ORDER BY valid_from)` with a `HAVING COUNT(*) >= n` filter within the time window

### CDC Integration

- [x] Integrate with the CDC replication log (introduced in v0.52.0): when a fact is asserted for a predicate registered in `_pg_ripple.temporal_predicates`, automatically set `valid_from = transaction_timestamp()`
- [x] When a temporal fact is retracted:
  - **`snapshot` model**: UPDATE the current open-ended row's `valid_to = transaction_timestamp()`; re-assertion creates a new row with `valid_from = now(), valid_to = NULL`
  - **`versioned` model**: never modifies existing rows; retraction closes the latest open row; re-assertion always INSERTs a new row — full immutable history preserved
- [x] `pg_ripple.temporal_cdc_enabled` GUC (BOOL, default `on`) — when `off`, CDC integration is bypassed and `valid_from` must be set manually (for historical data loading)

### Temporal Pattern Detection Datalog Rules

- [x] Parser support for the three new operators in Datalog rule bodies
- [x] Compile each operator to a parameterized CTE fragment that is inlined into the semi-naive evaluation loop
- [x] Example rule (clinical alerting):
  ```
  ex:consecutiveFever(?patient) :-
      CONSECUTIVE(3, ex:feverReading, 'P3D'),
      ?patient ex:feverReading ?temp,
      ?temp > 38.0 .
  ```

### Tests

- [x] `tests/pg_regress/sql/v0107_temporal_sequential.sql`:
  - Insert time-ordered temporal facts at known timestamps; assert `WITHIN` returns correct subjects
  - Insert pairs of events at known relative timestamps; assert `SEQUENCE` returns the correct subjects where event A precedes event B within the window
  - Insert `n` consecutive readings; assert `CONSECUTIVE` fires at `n` and not at `n-1`
  - **Window boundary edge cases**: fact exactly at window start, fact exactly at window end, zero-duration window
  - **CDC integration**: insert a fact via `insert_triple()` on a CDC-enabled temporal predicate; assert `valid_from` is approximately `transaction_timestamp()`
  - **`snapshot` retraction**: retract a fact; assert `valid_to` is set on the closed row and a new open row exists after re-assertion
  - **`versioned` retraction**: retract a fact; assert previous rows are unmodified and a new closed row is added
