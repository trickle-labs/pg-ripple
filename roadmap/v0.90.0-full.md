## 1. Correctness Fixes

### CB-02 — PageRank Convergence Norm Documentation

**Finding**: The convergence check uses L1 norm internally but this is not documented. Users porting from NetworkX (L1) or igraph (L2) cannot reason about behaviour differences.

**Implementation**:
1. Add to `docs/src/features/pagerank.md` under "Convergence":
   ```markdown
   ## Convergence Norm
   
   pg_ripple uses the **L1 norm** (sum of absolute differences) to test convergence
   between iterations, consistent with NetworkX's default behaviour. The convergence
   threshold `pg_ripple.pagerank_convergence_delta` (default 0.0001) is compared
   against the L1 norm of the score delta vector.
   
   Alternative norms can be selected via `pg_ripple.pagerank_convergence_norm` GUC:
   - `'l1'` (default): sum of |Δscore_i| — fastest to compute
   - `'l2'`: sqrt(sum of Δscore_i²) — matches igraph default
   - `'linf'`: max |Δscore_i| — most conservative (slowest convergence)
   ```
2. Register `pg_ripple.pagerank_convergence_norm` GUC in `src/gucs/registration/pagerank.rs`.
3. Wire norm selection in `src/pagerank/executor.rs` convergence check.

**Effort**: S.

### CB-04 — IVM K-hop Drift Bound

**Finding**: The bounded K-hop refresh approximation has no documented error bound and no automatic fallback when accumulated drift exceeds a threshold.

**Implementation**:
1. Documentation: `docs/src/features/pagerank.md` — "Incremental Refresh Error Bounds" section:
   ```markdown
   The theoretical error bound after K hops of incremental propagation is:
   |Δscore| ≤ α^K × max_delta_per_iteration
   where α is the damping factor (default 0.85) and K is `pagerank_khop_limit` (default 30).
   At K=30, α^30 ≈ 0.0076; the maximum error per dirty node is < 1% of the per-iteration delta.
   ```
2. New GUC `pg_ripple.pagerank_full_recompute_threshold` (FLOAT8, default 0.01): when the fraction of `stale = true` rows in `pagerank_scores` for a given topic exceeds this threshold, the next IVM worker cycle triggers a full `pagerank_run()` for that topic.
3. `_pg_ripple.pagerank_dirty_edges` gains a `stale_fraction` computed column in the stats view.

**Effort**: M.

## 2. Performance Improvements

### PERF-01 — WCOJ Integration for Large-Graph PageRank

**Finding**: PageRank per-iteration SQL hash join does not use the WCOJ executor for large graphs. For 100M+ edge graphs this is 10–100× slower than necessary.

**Implementation**:

The existing WCOJ executor in `src/sparql/wcoj.rs` implements the Leapfrog Triejoin algorithm. Integration with PageRank requires:

1. **PERF-01-THRESHOLD**: In `src/pagerank/executor.rs`, at `pagerank_run()` entry, query `_pg_ripple.predicates.triple_count` for the edge predicate(s). If total > `pg_ripple.pagerank_wcoj_threshold` (default 10M) AND `pg_ripple.wcoj_enabled = on`, select WCOJ scan path.

2. **PERF-01-SCAN**: The per-iteration neighbour scan:
   ```rust
   // Standard SQL path (< 10M edges):
   // SELECT s, o, w FROM vp_{pred_id} WHERE g = ?graph
   
   // WCOJ path (≥ 10M edges):
   // Uses wcoj_execute() with sorted VP arrays for leapfrog join
   // on the (s, o) index — no planner re-entry per iteration
   ```

3. **PERF-01-EXPLAIN**: `explain_inference()` gains `pagerank_scan_mode: 'wcoj' | 'sql'` field.

4. **PERF-01-BENCH**: `benchmarks/pagerank_scale.sh` reports wall time for 1M/10M/100M edge graphs with and without WCOJ. Baseline written to `benchmarks/pagerank_throughput_history.csv`.

**Effort**: L (largest item in v0.90.0; estimated 3–5 days).

### PERF-04 — PageRank Streaming Without Temp Materialisation

**Finding**: `pagerank_run()` writes the full edge set to PostgreSQL temp tables on every call when `pagerank_incremental = off`. For 100M edge graphs this is 4–8 GB.

**Implementation**:
1. Compute `estimated_edge_bytes = edge_count * 8` bytes.
2. Compare against `work_mem * pagerank_partitions` (GUC values).
3. If below threshold, stream directly from VP tables in the Datalog iteration loop without writing to temp.
4. New GUC `pg_ripple.pagerank_temp_threshold` (BIGINT, default auto from `work_mem`): explicit override.

The temp-table path is retained for graph-partitioned parallel evaluation (`pagerank_partition = on`) where each worker needs a stable snapshot.

**Effort**: M.

### CON-03 — Advisory Lock for pagerank_run()

**Implementation**:
```rust
// src/pagerank/executor.rs — pagerank_run() entry
let lock_key = format!("pagerank_run_{}", topic.unwrap_or(""));
client.execute(
    "SELECT pg_advisory_xact_lock(hashtext($1))",
    &[&lock_key]
```

Documentation added to `docs/src/features/pagerank.md`:

### DL-01 — Probabilistic Weight Parser Validation

`src/datalog/parser.rs` extended with weight validation:
```rust
match weight {
    w if w.is_nan() => pgrx::error!("PT0301 @weight(NaN) is not a valid probability"),
    w if w < 0.0 => pgrx::error!("PT0301 @weight({}) must be ≥ 0.0", w),
    w if w > 1.0 => pgrx::error!("PT0301 @weight({}) must be ≤ 1.0", w),
    _ => weight,
}
```

Proptest reject-case suite: verifies NaN, -0.001, 1.001, -∞, +∞ all raise PT0301.

### DL-02 — Cyclic Convergence Documentation

`docs/src/features/uncertain-knowledge.md` — new section "Convergence Guarantees for Cyclic Probabilistic Rules":

> When `prob_datalog_cyclic = on`, pg_ripple iterates the noisy-OR composition to fixpoint. The noisy-OR operator is **monotone** on [0, 1] (adding more evidence can only increase confidence, never decrease it), which guarantees that the semi-naive iteration sequence is non-decreasing and bounded above by 1.0. Therefore, fixpoint convergence is guaranteed for any finite probabilistic Datalog program with noisy-OR semantics.
>
> This result follows directly from Theorem 2 in De Raedt, Kimmig & Toivonen (2007), *ProbLog: A Probabilistic Prolog and its Application in Link Discovery*.
>
> Convergence speed depends on cycle depth and confidence values; programs with near-1.0 confidence in cycles may converge slowly. The `prob_datalog_max_iterations` GUC (default 100) and `prob_datalog_convergence_delta` GUC (default 1e-6) control termination.
