# v0.52.0 — pg-trickle Relay Integration

**Effort estimate: 5–7 person-weeks**

**Goal:** Connect pg_ripple to the outside world via pg-trickle's relay transport layer. This release builds the glue code that lets pg_ripple act as a semantic hub in a hub-and-spoke architecture — collecting data from external sources (Kafka, NATS, webhooks), enriching it through the knowledge graph, and distributing the enriched results back out to consumers. See [plans/pg_trickle_relay_integration.md](../plans/pg_trickle_relay_integration.md) for the full architecture.

---

## JSON → RDF via JSON-LD `toRdf` (symmetric with framing)

**Turn any JSON event into RDF triples with a single function call — using the W3C standard, not a bespoke mapping.**
Right now, if you want to load a JSON message from Kafka into the knowledge graph, you have to write a PL/pgSQL trigger that manually builds a string of N-Triples. This is repetitive and error-prone — every team ends up writing slightly different versions of the same boilerplate.

pg_ripple already exports RDF as JSON-LD (with framing). The inverse is also a W3C standard: the [JSON-LD 1.1 API](https://www.w3.org/TR/json-ld11-api/) defines **Expand** + **toRdf**, which together turn a JSON document plus a `@context` into N-Quads. v0.52.0 ships exactly that, so ingestion and export are symmetric and round-trippable: `frame(load(toRdf(expand(json)))) ≡ json` (modulo blank-node relabelling).

```sql
-- Convert a JSON order event into N-Triples using a registered context
PERFORM pg_ripple.jsonld_to_ntriples(
    NEW.payload,
    context => 'schema-org'   -- or an inline JSONB context, or a URL
);

-- Convenience: bolt on @id and @type if the source JSON lacks them
PERFORM pg_ripple.jsonld_to_ntriples(
    NEW.payload
        || jsonb_build_object('@id', 'https://example.org/order/' || (NEW.payload->>'order_id'))
        || jsonb_build_object('@type', 'https://schema.org/Order'),
    context => 'schema-org'
);
```

Because the heavy lifting is done by the `@context`, every common need — nested objects, RDF lists vs repeated predicates (`@container: @list`), XSD-typed values (`@type` coercion), key-to-IRI mapping, base IRIs — is expressed declaratively in standard JSON-LD instead of bespoke arguments. Contexts can be inlined per call, fetched from a URL, or registered server-side once via `pg_ripple.register_jsonld_context(name, document)` and looked up from `_pg_ripple.jsonld_contexts`.

For JSON shapes that no `@context` can reshape (one field becoming two triples, value transformations, conditional emission), PL/pgSQL custom triggers remain the escape hatch; deeper declarative options (RML / SPARQL-Generate) are deferred to a future release.

**Documentation** covers the JSON-LD context model, the registered-context registry, conformance against the [JSON-LD 1.1 toRdf test suite](https://w3c.github.io/json-ld-api/tests/), and how to call `jsonld_to_ntriples()` from pg-trickle stream table triggers.

## CDC → pg-trickle Outbox Bridge Worker

**Automatically write enriched triples to pg-trickle's outbox for relay distribution.**
pg_ripple already fires `NOTIFY` when new triples are inserted (including triples inferred by Datalog rules). In v0.54.0, a new background worker listens for those notifications, decodes the integer dictionary IDs back to human-readable IRIs and literals in bulk, and writes the decoded triples as JSON-LD events into a pg-trickle outbox-compatible stream table.

The worker is designed for throughput: it batches notifications rather than processing them one at a time, uses a single dictionary-decode query per batch, and writes to the outbox with a multi-row INSERT. Batch size and flush interval are configurable via GUC parameters (`pg_ripple.cdc_bridge_batch_size`, `pg_ripple.cdc_bridge_flush_ms`).

This makes the full "inbound event → enrichment → outbound relay" pipeline possible without any application code — just configuration:

```
Kafka → pg-trickle inbox → trigger → load_ntriples() → Datalog inference
→ CDC notification → bridge worker → pg-trickle outbox → relay forward → NATS
```

**Documentation** covers the bridge worker architecture, GUC tuning, and backpressure handling when the outbox grows faster than the relay drains it.

## Selective CDC Bridge Triggers

**Zero-latency outbound delivery for high-priority triples.**
The bridge worker (above) introduces a small delay — typically 50–500 ms — because it operates asynchronously. For high-priority data paths (for example, security alerts that must reach a webhook within 10 ms), this is too slow.

v0.54.0 adds configurable triggers on VP delta tables that write directly to a pg-trickle stream table within the same transaction, before the transaction even commits. These triggers are selective: you configure which predicates to watch (e.g., `ex:alertLevel`) and optionally a SPARQL FILTER expression, so only matching triples cause an immediate outbox write.

```sql
-- Trigger-based bridge for high-priority alerts (< 10 ms latency)
SELECT pg_ripple.enable_cdc_bridge_trigger(
    'alert-fast-path',
    predicate  => '<https://example.org/alertLevel>',
    outbox     => 'enriched_events'
);
```

Triggers and the bridge worker are complementary: use triggers for latency-sensitive paths and the worker for bulk enriched data.

## JSON-LD Event Serializer

**Efficient single-event and small-batch serialization for outbound relay payloads.**
pg_ripple already has `export_jsonld()` for exporting entire named graphs as JSON-LD. That function is optimised for bulk export, not for serializing one or two triples per event in a high-frequency stream.

v0.54.0 adds `pg_ripple.triple_to_jsonld(s BIGINT, p BIGINT, o BIGINT) RETURNS JSONB` — a thin, fast path that decodes a single triple from dictionary IDs and returns a JSON-LD object. It reuses the LRU dictionary cache so repeated accesses to common IRIs (class names, property names) require no database round-trips.

For small batches (e.g., a Datalog inference burst of 5–20 triples about the same subject), `triples_to_jsonld(subject BIGINT) RETURNS JSONB` collects all triples for that subject into a single JSON-LD document with a `@context` block.

Both functions are used internally by the bridge worker and CDC bridge triggers, and are also directly callable from SQL.

## Outbox Dedup Key from Statement ID

**Guarantee idempotent delivery even across relay restarts.**
pg_ripple assigns every triple a globally unique statement identifier (the `i` column in VP tables, drawn from a shared sequence). v0.54.0 exposes this as a relay-compatible dedup key:

```sql
SELECT pg_ripple.statement_dedup_key(s, p, o)  -- returns 'ripple:{statement_id}'
```

When this key is stored as the outbox `event_id`, the relay's dedup mechanism prevents the same triple from being delivered twice even if the relay restarts mid-batch. This is especially important for high-availability deployments where relay pods may fail and restart.

## Vocabulary Alignment Templates

**Pre-built Datalog rules for common JSON-to-RDF vocabulary mappings.**
When data arrives from different sources using their own field names, the knowledge graph ends up with fragmented vocabulary: CRM data uses `crm:customerName`, ERP data uses `erp:accountTitle`, but downstream consumers expect `schema:name`. Writing the alignment rules by hand is straightforward but repetitive.

v0.54.0 ships a library of pre-built Datalog rule templates for common vocabulary pairs:

| Source Vocabulary | Target Vocabulary | Rule Set |
|---|---|---|
| Schema.org | SAREF (IoT) | `vocab/schema_to_saref.pl` |
| Schema.org | FHIR (healthcare) | `vocab/schema_to_fhir.pl` |
| Schema.org | PROV-O (provenance) | `vocab/schema_to_provo.pl` |
| JSON-LD generic | Schema.org | `vocab/generic_to_schema.pl` |

Rules are loaded via `pg_ripple.load_rules_from_file()` and can be layered — for example, load CRM-specific alignment first, then the generic Schema.org rules. The templates serve as starting points and are expected to be customised per deployment.

## pg-trickle Runtime Detection & Graceful Degradation

**Bridge features work only when pg-trickle is installed; they degrade gracefully when it isn't.**
pg_ripple and pg-trickle are independent extensions. Some pg_ripple deployments will never use pg-trickle. v0.54.0 adds a runtime check at extension load time:

- If pg-trickle is present: all bridge functions (`enable_cdc_bridge_trigger()`, CDC bridge worker, outbox dedup key) are fully operational.
- If pg-trickle is absent: bridge functions return a clear error message (`PT800: pg_trickle extension is not installed`); the rest of pg_ripple is unaffected.
- GUC `pg_ripple.trickle_integration = off` disables the bridge explicitly, useful when both extensions are installed but the integration is not needed.

This means pg_ripple can be installed in any environment without requiring pg-trickle as a dependency.

## End-to-End Integration Test Suite

**Automated tests for the full relay pipeline.**
The integration with pg-trickle introduces a new class of behaviour — multi-extension, event-driven, asynchronous — that unit tests cannot fully cover. v0.54.0 adds an end-to-end test suite that exercises the complete pipeline:

```
Simulated source (SQL INSERT into inbox)
  → pg-trickle stream table trigger
  → pg_ripple.load_ntriples()
  → Datalog inference
  → CDC notification
  → Bridge worker (or CDC trigger)
  → pg-trickle outbox row
  → Captured and verified by test harness
```

The suite runs in CI using a PostgreSQL 18 instance with both extensions installed. Tests cover:
- Single-event round-trip latency (trigger path: < 20 ms; worker path: < 500 ms)
- Batch throughput (1,000 events/s sustained for 60 s)
- Dedup key uniqueness across relay restart simulation
- Graceful degradation when pg-trickle is uninstalled mid-run
- Backpressure: outbox growth is bounded when the relay is paused

## What This Enables

After v0.54.0, a complete hub-and-spoke knowledge graph pipeline is possible with only SQL configuration — no application code required:

```sql
-- 1. Install both extensions
CREATE EXTENSION pg_trickle;
CREATE EXTENSION pg_ripple;

-- 2. Inbound: Kafka → inbox → RDF
SELECT pgtrickle.set_relay_inbox('orders', inbox => 'order_inbox',
    source => '{"type":"kafka","topic":"orders"}');

-- 3. Transform: JSON → triples via JSON-LD toRdf (using v0.52.0 helper)
SELECT pg_ripple.register_jsonld_context('schema-org', $$ { "@vocab": "https://schema.org/" } $$);
CREATE TRIGGER order_to_rdf AFTER INSERT ON order_inbox
FOR EACH ROW EXECUTE FUNCTION pg_ripple.jsonld_to_ntriples_trigger(
    context_name => 'schema-org',
    id_template  => 'https://example.org/order/{order_id}',
    type_iri     => 'https://schema.org/Order');

-- 4. Enrich: Datalog inference + SHACL validation (existing)
SELECT pg_ripple.load_rules('enrichment', $$ ... $$);
SELECT pg_ripple.load_shacl($$ ... $$);

-- 5. Outbound: enriched triples → outbox → relay → NATS
SELECT pg_ripple.enable_cdc_bridge_trigger('alerts',
    predicate => '<ex:highValueCustomer>', outbox => 'enriched_events');
SELECT pgtrickle.set_relay_outbox('enriched-to-nats',
    outbox => 'enriched_events', sink => '{"type":"nats",...}');
```

> **Note (v0.93.0)**: The `pgtrickle.set_relay_inbox` and `pgtrickle.set_relay_outbox` examples
> above apply to pg_trickle \u2264 v0.45.0. Starting from pg_trickle v0.46.0, relay features
> were extracted to the `pg_tide` extension (`trickle-labs/pg-tide`). For v0.46.0+, use
> `tide.relay_set_inbox(...)` and `tide.relay_set_outbox(...)` instead.
> See [pg-trickle-relay.md](../docs/src/operations/pg-trickle-relay.md) for updated examples.
