# v0.117.0 — A16 Low-Severity Polish, Tests and Supply-Chain (Full Details)

**Status: Planned** | **Scope: Medium**

## Overview

v0.117.0 completes the Assessment 16 remediation cycle. After v0.112.0 (critical
and high), v0.113.0 (high + performance), v0.114.0 (module splits), v0.115.0
(HTTP API + observability), and v0.116.0 (correctness GUCs + cache bounds), this
release addresses the remaining 15 low-severity findings and two deferred medium
items (concurrency tests and fuzz targets).

The release is intentionally narrow: documentation, CI hygiene, supply-chain
hardening, and test coverage. No schema changes. No new user-visible API.

> **Effort estimate: 2–3 person-weeks**

### L16-01 — `#[allow(...)]` Suppression Reduction

**Target:** ≤ 186 suppressions (10% reduction from the 206 counted in Assessment 16).

Approach:
1. `grep -rn '#\[allow(' src/ | wc -l` — establish baseline
2. For each suppression, determine if it can be replaced with a targeted fix
3. Suppressions that cannot be fixed must be annotated: `// A16-CQ: <reason>`

Common fixable patterns:
- `#[allow(dead_code)]` on items that should be `pub(crate)` or deleted
- `#[allow(clippy::too_many_arguments)]` — refactor into a `struct` parameter

### L16-05 — Arrow Flight v1→v2 HMAC Ticket Migration

The Arrow Flight endpoint (introduced in v0.79.0) uses HMAC-signed tickets for
bulk export authorisation. v0.91.0 changed the HMAC scheme from SHA-256 (v1) to
SHA-512 (v2). The migration window was defined as 90 days.

`docs/src/reference/arrow-flight.md` will document:

| Ticket Version | HMAC Algorithm | Valid Until |
|----------------|----------------|-------------|
| v1 | HMAC-SHA-256 | 2026-09-01 (90 days after v0.91.0 release) |
| v2 | HMAC-SHA-512 | Current |

To identify an expired v1 ticket:
```bash
# Ticket format: base64(version_byte || hmac_bytes || payload)
# version_byte = 0x01 for v1, 0x02 for v2
echo "$TICKET" | base64 -d | head -c 1 | xxd
# 0x01 = v1 (expired), 0x02 = v2 (current)
```

### L16-12 — SBOM Signing with Cosign

`release.yml` addition (after SBOM generation step):

```yaml
- name: Sign SBOM with cosign
  uses: sigstore/cosign-installer@v3
  with:
    cosign-release: 'v2.2.4'
- run: |
    cosign sign-blob \
      --bundle pg_ripple.cdx.json.bundle \
      pg_ripple.cdx.json
- name: Upload signed SBOM
  uses: actions/upload-artifact@v4
  with:
    name: sbom-signed
    path: |
      pg_ripple.cdx.json
      pg_ripple.cdx.json.bundle
```

Verification instruction in `RELEASE.md`:
```bash
cosign verify-blob \
  --bundle pg_ripple.cdx.json.bundle \
  pg_ripple.cdx.json
```

### M16-13 — Fuzz Target Additions

New fuzz targets follow the existing pattern in `fuzz/fuzz_targets/`:

**`temporal_query.rs`:** Fuzz `point_in_time()` and `mark_temporal()` with
arbitrary timestamp strings and IRI inputs — targeting parse panics.

**`pprl_bloom_encode.rs`:** Fuzz `bloom_encode()` with arbitrary value strings,
key bytes, and hash-count/filter-size parameters — targeting integer overflow in
Bloom filter sizing.

**`rule_authoring_validate.rs`:** Fuzz the Datalog rule parser with arbitrary
rule text — targeting infinite loops in the grammar and stack overflows in the
stratifier.

**`skos_bundle.rs`:** Fuzz bundle activation with arbitrary bundle name strings —
targeting SQL injection in `load_datalog_bundle()` (should have no effect after M16-04
fixes but provides regression coverage).
