# v0.95.0 — A15 Medium: Correctness, Security, Storage (Full Details)

**Version**: 0.95.0
**Milestone**: Assessment 15 Medium Remediation
**Status**: Planned
**Scope**: Large

## Background

This release addresses the medium-priority items from Assessment 15 that relate to correctness, security, and storage. Previous releases addressed the high-priority items (v0.94.0).

## Deliverables

### M15-01: Replace `unreachable!()` panics with user-visible errors

**Files**: `src/pagerank/export.rs`, `src/pagerank/centrality.rs`

Both files contain `_ => unreachable!()` match arms that would produce a PostgreSQL server crash (panic → process abort) if triggered by an unexpected input. Since the callers already guard the format string, these paths should be defense-in-depth `pgrx::error!` calls that produce a clean error message rather than a crash.

**CI gate**: `scripts/check_no_unreachable_in_production.sh` added to CI under the `lint` job. Scans `src/pagerank/` and `src/storage/` for bare `unreachable!()` usage and fails if any are found.

### M15-02: Resolve-once DNS rebinding fix

**File**: `src/sparql/federation/policy.rs`

The current `check_endpoint_policy()` validates the URL's hostname at policy-check time but the actual HTTP connection is made later, potentially to a different IP if the DNS record changed between the two calls (DNS rebinding / TOCTOU vulnerability).

The fix: resolve the hostname **once**, validate the resolved IP against the SSRF blocklist, and return the resolved IP alongside the validation result so the HTTP client can connect to the IP directly with a pinned `Host:` header.

**New function**: `check_endpoint_policy_resolved(url: &str) -> Result<ResolvedEndpoint, String>` where `ResolvedEndpoint` carries `{ url, resolved_ip, host_header }`.

### M15-03: `sql_drop` event trigger for replication-slot cleanup

**File**: `src/schema/triggers.rs`

When users run `DROP EXTENSION pg_ripple`, any active CDC logical-replication slots (e.g. `pg_ripple_cdc`) remain in `pg_replication_slots` and continue consuming WAL, eventually causing PostgreSQL to run out of disk space.

A new `sql_drop` event trigger `_pg_ripple_cleanup_on_drop` detects `DROP EXTENSION pg_ripple` (by inspecting `pg_event_trigger_dropped_objects()` for `object_type = 'extension'`) and calls `pg_drop_replication_slot()` for each slot with `plugin = 'pg_ripple'`.

### M15-04: `redacted_error()` for SSE initialisation errors

**File**: `pg_ripple_http/src/stream.rs`

The SSE stream handler in `stream_sparql_select()` produces raw inline JSON error responses that expose internal error details to the client. These should use `redacted_error()` from `common.rs` which logs the full error internally and returns only a trace-ID to the client.

### M15-07: Scheduled `VACUUM ANALYZE _pg_ripple.dictionary` after bulk encode

**Files**: `src/gucs/storage.rs`, `src/gucs/registration/storage.rs`, `src/dictionary/mod.rs`

After large bulk dictionary encodes (many new terms), the dictionary table's autovacuum may not run quickly enough, causing the planner to work with stale statistics. This adds:

1. `pg_ripple.dict_vacuum_threshold` GUC (default: 10000): when a batch encode inserts more than this many new terms, run `VACUUM ANALYZE _pg_ripple.dictionary` immediately.
2. `autovacuum_scale_factor = 0.01` storage parameter on the dictionary table (so autovacuum fires more aggressively for this high-churn table).

### M15-08: Regression tests for OPTIONAL + property paths + vp_rare + GRAPH

**File**: `tests/pg_regress/sql/sparql_optional_path_in_graph_rare.sql` (+ expected output)

Adds explicit regression tests covering the combination:
- `OPTIONAL {}` inside `GRAPH <g> {}`
- Property path expressions (`+`, `*`, `/`, `|`)
- Predicates stored in `vp_rare` (low-frequency predicates)

This combination was identified as a gap where interactions between the rare-predicate view, graph filtering, OPTIONAL join, and property path CTEs could produce incorrect results.

### M15-09: NaN/Inf/out-of-range rejection for confidence values

**File**: `src/bulk_load.rs`

The current range check `!(0.0..=1.0).contains(&confidence)` implicitly rejects NaN and Inf (because NaN comparisons always return false), but the error message says nothing about NaN or Inf specifically. Add explicit checks:

```rust
if confidence.is_nan() {
    pgrx::error!("confidence must be a finite number in [0.0, 1.0]; got NaN (PT0301)");
}
if confidence.is_infinite() {
    pgrx::error!("confidence must be a finite number in [0.0, 1.0]; got {:?} (PT0301)", confidence);
}
```

Also add the same validation to the `INSERT ON CONFLICT` confidence path in the direct SQL API.

### M15-10: Fold `schema_generation` counter into plan cache key

**Files**: `src/schema/`, `src/sparql/plan_cache.rs`, `src/storage/mod.rs`, `src/storage/promote.rs`

When VP tables are promoted or new predicates are registered via `ensure_vp_table()`, the SQL plan generated for a SPARQL query may become invalid (e.g. a query that previously used `vp_rare` must now use a dedicated VP table). The plan cache must be invalidated in these cases.

**Solution**: Add `_pg_ripple.schema_generation_seq` sequence. Increment it in `ensure_vp_table()` and `promote_predicate_impl()`. Include the current value in the plan cache key so any VP schema change automatically invalidates cached plans.

### M15-12: ADD/COPY/MOVE through main UPDATE pipeline

**File**: `src/sparql/execute/mod.rs`

The `try_execute_add_copy_move()` function returns early from `sparql_update()` before the mutation-journal flush and audit log entries are written. This means:
- CDC subscribers do not see ADD/COPY/MOVE changes
- SHACL validation queue is not triggered
- SPARQL audit log has no entry for these operations

Fix: apply the same post-processing (mutation journal flush, audit log) after ADD/COPY/MOVE operations.
