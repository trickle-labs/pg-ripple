# v0.106.0 — Temporal Reasoning Phase 1: Temporal Fact Store & Basic Operators

**Theme**: Store facts with time intervals and query them with temporal operators.

> **In plain language:** Most knowledge changes over time. A patient's diagnosis is valid from
> the date it was made until it was revised. An employee's job title is valid from their
> promotion date until their next title change. A contract is valid between its start and
> expiry dates. Standard RDF triples have no time dimension — they're just facts that either
> exist or don't. This release adds a first-class temporal layer: mark any predicate as
> "time-aware", and pg_ripple will record when each fact was valid. Then ask questions like
> "which facts held in January 2025?" or "which drug prescriptions were active during this
> hospital visit?" using natural temporal operators.
>
> **Foundation document**: [plans/expert-system.md](../plans/expert-system.md)
>
> **Effort estimate: 3–4 person-weeks**

## Prerequisites

- v0.99.2 released and all tests passing.
- Independent of v0.100.0–v0.105.0 (Expert System phases 1–6) — can ship in parallel.

## Deliverables

### Temporal Fact Store

**No changes to existing VP table schemas.** Temporal facts live in a dedicated table:

- [ ] `_pg_ripple.temporal_facts (s BIGINT, p BIGINT, o BIGINT, g BIGINT, valid_from TIMESTAMPTZ NOT NULL, valid_to TIMESTAMPTZ)` — one row per (fact, validity interval); `valid_to IS NULL` means the fact is currently valid (open-ended interval)
- [ ] Indexes:
  - B-tree on `(s, p, valid_from, valid_to)` for subject-scoped temporal queries
  - B-tree on `(p, valid_from, valid_to)` for predicate-scoped temporal scans
  - Partial B-tree on `(valid_from, valid_to) WHERE valid_to IS NULL` for currently-valid facts

### Temporal Predicate Registry

- [ ] `_pg_ripple.temporal_predicates (predicate_id BIGINT PRIMARY KEY, data_model TEXT CHECK (data_model IN ('snapshot', 'versioned')))` — marks which predicates are time-aware and what model they use
- [ ] `pg_ripple.mark_temporal(predicate_iri TEXT, data_model TEXT DEFAULT 'snapshot') → VOID` — registers a predicate as temporal; raises PT0430 if already registered with a different model
- [ ] `pg_ripple.unmark_temporal(predicate_iri TEXT) → VOID` — unregisters; raises PT0431 if existing temporal facts still exist for this predicate (must be deleted first)
- [ ] `pg_ripple.temporal_data_model` GUC (`'snapshot'` | `'versioned'`, default `'snapshot'`) — default model for `mark_temporal()` when `data_model` is omitted

**Data models:**
- `snapshot`: each fact has at most one currently-open interval; asserting a new value closes the previous one
- `versioned`: every assertion always creates a new row; retraction closes the latest open row without modifying previous rows; full version history is preserved

### Query Layer Integration

- [ ] The query layer routes temporal predicates to `temporal_facts`; atemporal predicates continue using VP tables — routing is transparent to callers but requires a predicate-type check in `sparql/plan.rs` (SPARQL BGP compilation) and `datalog/seminaive.rs` (Datalog rule evaluation) that consults `_pg_ripple.temporal_predicates` before generating the VP table scan
- [ ] `pg_ripple.enable_temporal_operators` GUC (BOOL, default `off`) — gates the temporal parse extensions; must be `on` to use `AFTER`, `BEFORE`, `DURING` in Datalog

### Temporal Data Insertion

- [ ] `pg_ripple.insert_triple_temporal(subject TEXT, predicate TEXT, object TEXT, valid_from TIMESTAMPTZ, valid_to TIMESTAMPTZ DEFAULT NULL, graph TEXT DEFAULT NULL) → BIGINT` — inserts a temporal fact directly; raises PT0432 if `predicate` is not registered as temporal
- [ ] For `snapshot` model: if an open-ended row already exists for the same `(s, p, o, g)`, closes it by setting `valid_to = valid_from` of the new row before inserting
- [ ] For `versioned` model: always inserts a new row regardless of existing rows
- [ ] PT0432: `insert_triple_temporal: predicate '%s' is not registered as temporal — call mark_temporal() first`

### Temporal Operators in Datalog

- [ ] `AFTER(timestamp)` built-in predicate — matches temporal fact rows where `valid_from > timestamp`
- [ ] `BEFORE(timestamp)` — matches rows where `valid_from < timestamp`
- [ ] `DURING(from_ts, to_ts)` — matches rows where `tsrange(valid_from, valid_to) && tsrange(from_ts, to_ts)`
- [ ] All three accept either `xsd:dateTime` literal arguments or Datalog variables bound to datetime values

### `pg:temporal_window()` SPARQL Function

- [ ] `pg:temporal_window(?subject, ?predicate, ?start, ?end)` SPARQL FILTER function — evaluates to true if a temporal fact for `(?subject, ?predicate, ?)` exists with a validity interval overlapping `[?start, ?end]`

### `sh:validFor` SHACL Constraint

- [ ] `sh:validFor "P1Y"^^xsd:duration` SHACL constraint — validates that no temporal fact for the constrained predicate has a `valid_to - valid_from` interval exceeding the XSD duration; violations reported via the standard SHACL validation pipeline

### Error Catalog

- [ ] PT0430: `mark_temporal: predicate '%s' is already registered with data model '%s'`
- [ ] PT0431: `unmark_temporal: predicate '%s' has %d existing temporal facts — delete them first`
