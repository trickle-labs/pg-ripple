# v0.73.0 - SPARQL 1.2 Tracking, Live Subscriptions, and Ecosystem Hardening

> **Full technical details:** [v0.73.0-full.md](v0.73.0-full.md)

**Status: Planned** | **Scope: Large**

> v0.73.0 addresses the Dimension 12 ecosystem gaps from Assessment 10: SPARQL 1.2 design tracking, a live query subscription API, and a set of ecosystem hardening items (CONTRIBUTING.md, Helm chart SHA pinning, feature status taxonomy docs, and `src/r2rml.rs` virtual-vs-materialized clarification).

---

## What is this?

Assessment 10 Dimension 12 identified several missing ecosystem capabilities that, while individually small, matter for production adoption. This release bundles the SPARQL 1.2 design-tracking document, a WebSocket/SSE live subscription API prototype, and a round of ecosystem hygiene items into a single focused delivery before v1.0.0.

## Headline outcomes

- `plans/sparql12_tracking.md` design document captures the SPARQL 1.2 delta (RDF-star integration, `LATERAL`, `VALUES` extensions, updated built-ins) and a compatibility assessment.
- `pg_ripple.subscribe_sparql(query TEXT)` SQL function and `pg_ripple_http /subscribe` WebSocket/SSE endpoint prototype: sends a notification whenever a SPARQL SELECT query over any affected graph would produce a different result.
- `docs/src/reference/feature-status-taxonomy.md` documents the promotion criteria for each status value in `feature_status()`.
- `CONTRIBUTING.md` added at the repository root with branch, commit, clippy, and pg_regress conventions.
- Helm chart in `charts/pg_ripple/` pins `pg_ripple_http` image to a release SHA instead of `latest`.
- `src/r2rml.rs` scope documented: the module materializes triples (it is not a virtual graph layer); a future virtual R2RML graph layer is tracked in `plans/r2rml-virtual.md`.
- `src/llm/` and `src/kge.rs` modules added to `feature_status()` with accurate status values.
- `pg_ripple.control` `comment` field updated to describe v0.73.0 highlights.
- New `pg_ripple.json_ld_load(document, default_graph)` SQL function: ingests a full JSON-LD document with `@graph`, walking each top-level node under its declared `@id`. Closes the asymmetry between inbound (single-subject only) and outbound (multi-graph supported) JSON-LD paths.
- New `pg_ripple.register_json_mapping(name, context, shape)` SQL function: stores a named bidirectional JSON↔RDF mapping. The engine derives both the ingest context map and the outbound frame template from a single registered object, eliminating the duplication of writing the same IRI–term correspondence twice. When a SHACL shape is referenced, nested node structure and datatype constraints are derived automatically from `sh:property` / `sh:node` without a separate frame; SHACL and mapping are validated for consistency at registration time.
