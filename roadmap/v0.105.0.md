# v0.105.0 — Guided Rule Authoring & LLM Rule Extraction

**Theme**: Write Datalog rules in plain English; let the LLM draft them for you.

> **In plain language:** The single biggest barrier to adopting a rule-based reasoning system is
> writing the rules. You need to know Datalog syntax, understand the knowledge graph's schema,
> and think carefully about edge cases. This release removes that barrier: you describe a rule
> in plain English — "Flag any two supplier records that share a VAT number as likely duplicates"
> — and pg_ripple generates candidate Datalog rules for you to choose from. You pick the one
> that looks right, validate it (the system will tell you about syntax errors, unused variables,
> and stratification issues), and load it. The LLM writes the first draft; the domain expert
> reviews and approves.
>
> **Foundation document**: [plans/expert-system.md](../plans/expert-system.md)
>
> **Effort estimate: 2–3 person-weeks**

## Prerequisites

- v0.104.0 released and all tests passing.
- `pg_ripple.llm_endpoint` configured (functions degrade gracefully when not set).

## Deliverables

### `draft_rule_from_nl()` SQL Function

- [ ] `pg_ripple.draft_rule_from_nl(description TEXT, candidates INT DEFAULT 3) → TABLE(rank INT, rule TEXT, explanation TEXT)`
  - Sends `description` to the LLM endpoint with a system prompt that includes: the current schema summary (`pg_ripple.schema_summary()`), available predicates, the Datalog syntax guide, and a few-shot example set of rule descriptions paired with correct Datalog rules
  - Returns the top `candidates` candidate rules ranked by the LLM's confidence; each row has the Datalog rule text and a one-sentence explanation of what the rule does
  - `candidates` must be in [1, 10] — raises PT0457 outside this range
  - When `llm_endpoint` is not configured, raises PT0458 (informative error, not a panic)
  - The returned rules are **not loaded** — caller must explicitly call `load_rules()` after review

### `validate_rule()` SQL Function

- [ ] `pg_ripple.validate_rule(rule TEXT) → JSONB`
  - Checks the rule without loading it into any rule set
  - Returns `{"valid": true}` or `{"valid": false, "errors": [...], "warnings": [...]}` where each error/warning has `"code"` and `"message"` fields
  - Errors (rule cannot be used): syntax errors, unbound head variables, unsafe negation
  - Warnings (rule may behave unexpectedly): unused body variables, rules that can never fire given the current schema, potential stratification issues, rules that would always derive the empty set

### `suggest_rules()` SQL Function

- [ ] `pg_ripple.suggest_rules(graph_iri TEXT, examples JSONB DEFAULT NULL) → TABLE(rule TEXT, support BIGINT, explanation TEXT)`
  - **Marked experimental** — API may change; results require domain expert validation before committing
  - Scans `graph_iri` for statistical co-occurrence patterns in the VP tables: pairs of predicates that frequently appear on the same subject, predicates whose values are often shared across subjects, etc.
  - Proposes candidate Datalog rules for each pattern with `support` = count of (subject, object) pairs matching the pattern
  - `examples` (optional): `[{"s": "...", "p": "...", "o": "..."}]` — example triples that the suggested rules should be able to derive; used to filter and rank candidates
  - Returns at most `pg_ripple.suggest_rules_max_candidates` (GUC, INT, default: 20) candidates to avoid overwhelming the user

### Error Catalog

- [ ] PT0457: `draft_rule_from_nl: candidates must be between 1 and 10, got %d`
- [ ] PT0458: `draft_rule_from_nl: pg_ripple.llm_endpoint is not configured`

### REST Endpoint

- [ ] `POST /rules/draft` in `pg_ripple_http` — request body: `{"description": "...", "candidates": 3}`, response: `[{"rank": 1, "rule": "...", "explanation": "..."}]`
- [ ] `POST /rules/validate` in `pg_ripple_http` — request body: `{"rule": "..."}`, response: `{"valid": true|false, "errors": [...], "warnings": [...]}`

### Tests

#### LLM Quality Test Strategy via Proptest

- [ ] `tests/proptest/rule_authoring.rs`:
  - Generates a random knowledge base with a known ground-truth rule (e.g. blocking rule: `candidate(?x, ?y) :- ?x ex:category ?c, ?y ex:category ?c, ?x != ?y .`)
  - Constructs a natural-language description of the rule from a template
  - Calls `draft_rule_from_nl()` with `candidates = 5`
  - Asserts at least one of the 5 candidates is semantically equivalent to the ground-truth rule: loads each candidate, runs inference against the same knowledge base, checks that the derived triples match the expected output
  - Runs against a local mock LLM that returns templated responses based on keywords in the description

#### Standard pg_regress Tests

- [ ] `tests/pg_regress/sql/v0105_rule_authoring.sql`:
  - `validate_rule()` on a syntactically correct rule → `{"valid": true}`
  - `validate_rule()` on a rule with an unbound head variable → `{"valid": false, "errors": [...]}`
  - `validate_rule()` on a rule with a stratification issue → `{"valid": true, "warnings": [...]}`
  - `suggest_rules()` on a small known graph → at least one returned candidate
  - PT0458 is raised when `llm_endpoint` is empty and `draft_rule_from_nl()` is called

## Exit Criteria

- `cargo pgrx regress pg18` passes all tests including `v0105_rule_authoring`
- `validate_rule()` correctly identifies syntax errors, unbound variables, and stratification warnings
- `suggest_rules()` returns at least one candidate on a non-trivial test graph
- PT0457 is raised when `candidates` is out of range
- PT0458 is raised when `llm_endpoint` is not configured
- `POST /rules/draft` and `POST /rules/validate` REST endpoints return valid JSON
