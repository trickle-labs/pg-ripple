# v0.120.0 — Operational & Ecosystem Features (Full Details)

**Status: Planned** | **Scope: Large**

## Overview

v0.120.0 completes the Platform Maturity phase (v0.118.0–v0.120.0). After the
temporal reasoning and circuit breaker work in v0.118.0–v0.119.0, this release
focuses on the operational surface: making pg_ripple easier to operate, monitor,
and integrate at scale. The headline items are rule-library federation (enabling
a marketplace of shared Datalog rule sets between instances) and read-replica
routing (enabling horizontal read scaling for SPARQL queries).

> **Effort estimate: 4–5 person-weeks**

## Feature 7 — PageRank Explain HTTP Endpoint

### Background

`explain_pagerank()` was added in v0.84.0 as a SQL function returning the top
contributors to a node's PageRank score. Dashboards and support workflows need
this without raw SQL access.

### Response Format

```json
{
  "node": "https://example.org/Alice",
  "score": 0.0342,
  "method": "personalized",
  "top_contributors": [
    { "source": "https://example.org/Bob", "contribution": 0.0121, "edge_count": 3 },
    { "source": "https://example.org/Carol", "contribution": 0.0089, "edge_count": 1 }
  ],
  "computed_at": "2024-01-15T10:23:45Z"
}
```

### IRI Encoding

IRIs in the path segment are percent-encoded. The handler uses:
```rust
let node_iri = urlencoding::decode(node_param)
    .map_err(|_| StatusCode::BAD_REQUEST)?;
```

The `urlencoding` crate is already a dependency via the existing SPARQL handlers.

## Feature 8 — Diagnostic Snapshot

### Background

Support tickets for pg_ripple typically require: extension version, GUC values,
table statistics, and recent metrics. Today this requires manual SQL queries.
The diagnostic snapshot bundles everything into one HTTP call.

### Response Structure

```json
{
  "snapshot_at": "2024-01-15T10:23:45Z",
  "extension_version": "0.120.0",
  "http_version": "0.120.0",
  "table_counts": {
    "_pg_ripple.dictionary": 1234567,
    "_pg_ripple.predicates": 89,
    "_pg_ripple.rules": 142
  },
  "gucs": {
    "pg_ripple.dictionary_cache_size": "64MB",
    "pg_ripple.vp_promotion_threshold": 1000,
    "pg_ripple.bulk_load_use_copy": true
  },
  "metrics_snapshot": "# HELP pg_ripple_triple_count ...\n..."
}
```

**Excluded GUCs** (sensitive):
- `pg_ripple.llm_api_key`
- `pg_ripple.arrow_flight_hmac_key`
- Any GUC ending in `_secret`, `_key`, `_token`, `_password`

GUC exclusion uses a suffix allow-list checked at snapshot time, not a hard-coded
list, to handle future GUCs correctly.

## Feature 11 — Rule-Library Federation

### Background

Organisations running multiple pg_ripple instances (e.g., one per team or department)
currently cannot share Datalog rule sets without manual copy-paste or Git-based
workflows. Rule-library federation makes rule sets first-class distributed artifacts,
publishable and subscribable via Arrow Flight.

### Architecture

```
Instance A                          Instance B
─────────                          ─────────
pg_ripple.publish_rule_library()   pg_ripple.subscribe_rule_library()
  → stores in _pg_ripple.rule_libs   → fetches from A's Flight stream
  → serves GET /rule-libraries/      → validates signature
    {name}/stream (Arrow Flight)     → installs locally via store_rules()
```

### SQL API

```sql
-- Publish a named rule library (makes it available via HTTP Flight endpoint)
SELECT pg_ripple.publish_rule_library(
  'my-ontology-rules',
  'http://localhost:8080/rule-libraries/my-ontology-rules/stream'
);

-- Subscribe to a remote rule library
SELECT pg_ripple.subscribe_rule_library(
  'http://other-instance:8080/rule-libraries/my-ontology-rules/stream',
  'my-ontology-rules'
);

-- The rules from the remote library are now active locally
SELECT count(*) FROM _pg_ripple.rules WHERE source_library = 'my-ontology-rules';
```

### Security

The Flight stream uses the existing HMAC ticket mechanism (introduced in v0.79.0,
upgraded to SHA-512 in v0.91.0). The subscriber must present a valid ticket issued
by the publisher. Rule content is signed with the publisher's HMAC key to prevent
tampering in transit.

## Feature 12 — Read-Replica Routing

### Background

SPARQL SELECT, CONSTRUCT, and ASK queries are read-only and safe to route to a
PostgreSQL streaming replica. For read-heavy workloads (dashboards, reporting),
this enables horizontal scaling without sharding.

### Configuration

```bash
# .env
PG_RIPPLE_HTTP_REPLICA_DSN=postgres://user:pass@replica-host:5432/mydb
```

### Routing Logic

```
Request: GET /sparql?query=SELECT+...&replica=ok

1. Check: is PG_RIPPLE_HTTP_REPLICA_DSN set?
   No → route to primary (as before)
   Yes → check replica health (cached, refreshed every 10s)

2. If replica healthy → proxy to replica connection pool
   If replica unhealthy → fall back to primary, log WARNING

3. Log: DEBUG "routing SPARQL SELECT to replica"
```

The `?replica=ok` parameter is explicit opt-in. It is never set automatically.
Write operations (UPDATE, INSERT via SPARQL UPDATE) always route to primary
regardless of the parameter.

### Connection Pool

A separate `bb8` connection pool is maintained for the replica DSN, with the
same pool size configuration as the primary (driven by
`PG_RIPPLE_HTTP_DB_POOL_SIZE`, default 10).

## Feature 9 — `just generate-helm-values` Recipe

### Usage

```bash
just generate-helm-values TENANT=acme
# Queries _pg_ripple.tenants WHERE name = 'acme'
# Emits: values-acme.yaml

helm install pg-ripple-acme ./charts/pg_ripple \
  --values values-acme.yaml
```

### Output Format

```yaml
# Generated by just generate-helm-values TENANT=acme on 2024-01-15
tenant:
  name: acme
  quotas:
    maxTriples: 10000000
    maxQueryMs: 30000
  graphs:
    - https://acme.example.org/data
```
