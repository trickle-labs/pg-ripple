# v0.101.0 — Natural Language Explanation

**Theme**: Plain-English explanations of why pg_ripple derived any fact.

> **In plain language:** v0.100.0 made pg_ripple record *why* it believes things — which rules
> fired, which base facts supported them — as a machine-readable JSON proof tree. This release
> takes that proof tree and turns it into plain English. Ask pg_ripple "why do you think Alice
> works at Acme Corp?" and it narrates the full chain of reasoning: "I know Alice works at
> Acme Corp because rule 'employment-graph' fired — it sees that Alice's employee ID exists in
> the HR graph and maps to the Acme Corp entity." Domain experts, auditors, and regulators can
> read the answer without understanding Datalog, SPARQL, or proof trees.
>
> **Foundation document**: [plans/expert-system.md](../plans/expert-system.md)
>
> **Effort estimate: 2–3 person-weeks**

## Prerequisites

- v0.100.0 released and all tests passing (`_pg_ripple.derivations` table available, `justify()` working).

## Deliverables

### `explain_inference()` SQL Function

- [ ] `pg_ripple.explain_inference(subject TEXT, predicate TEXT, object TEXT, format TEXT DEFAULT 'text') → TEXT`
  - Retrieves the proof tree from `_pg_ripple.derivations` via `justify(subject, predicate, object)`
  - Decodes all dictionary IDs to human-readable IRI strings and literal values
  - Builds a structured prompt and sends it to the LLM endpoint (`pg_ripple.llm_endpoint`) with a domain-appropriate system prompt: "You are explaining why a knowledge graph derived a fact. Given this proof tree, write a clear, concise explanation in plain English. Audience: domain expert, not a programmer."
  - Returns the LLM's narrative as plain text (`format = 'text'`) or as Markdown with embedded code blocks for the raw rule text (`format = 'markdown'`)
- [ ] `pg_ripple.explain_inference_jsonb(subject TEXT, predicate TEXT, object TEXT) → JSONB`
  - Returns `{"proof_tree": <justify() output>, "narrative": "<LLM explanation>"}` — the structured proof with embedded narrative for programmatic consumers

### LLM Unavailability Fallback

- [ ] When `pg_ripple.llm_endpoint` is not configured or the endpoint returns an error, `explain_inference()` falls back to a deterministic renderer:
  - Renders the proof tree as indented text: rule names, antecedent facts at each level, base facts at the leaves
  - Never returns an error — always returns something readable
  - Fallback is documented in the function's SQL comment and the docs chapter

### Explanation Cache

- [ ] `_pg_ripple.explanation_cache (sid BIGINT, format TEXT, model TEXT, explanation TEXT, cached_at TIMESTAMPTZ)` table — one row per (fact, format, LLM model) combination
- [ ] `pg_ripple.explanation_cache_ttl` GUC (INTERVAL, default `'1 hour'`) — explanations older than this are regenerated on next call
- [ ] `pg_ripple.vacuum_explanation_cache() → BIGINT` — removes expired rows; callable by the DBA or scheduled via `pg_cron`
- [ ] Cache is keyed on the SID of the derived fact, not the raw IRI strings, so dictionary churn does not create stale cache entries

### REST Endpoint

- [ ] `POST /explain` in `pg_ripple_http` — request body: `{"subject": "...", "predicate": "...", "object": "...", "format": "text|markdown"}`, response: `{"explanation": "...", "cached": false}`
- [ ] `GET /explain?subject=...&predicate=...&object=...` convenience alias for browser/curl use

### Tests

- [ ] pg_regress test `tests/pg_regress/sql/v0101_explain_inference.sql`:
  - Load a small knowledge base, enable `record_derivations`, run `infer('default')`
  - With a mock LLM endpoint (empty `llm_endpoint` GUC): assert `explain_inference()` returns non-empty fallback text containing the rule name
  - Assert `explain_inference_jsonb()` contains both `proof_tree` and `narrative` keys
  - Assert `explain_inference()` on a base (non-inferred) fact returns `null`
- [ ] Mock-LLM test coverage for the error path (unreachable endpoint → fallback, not error)

### Migration & Control

- [ ] Migration script `sql/pg_ripple--0.100.0--0.101.0.sql` — creates `_pg_ripple.explanation_cache`
- [ ] `pg_ripple.control` updated to `default_version = '0.101.0'`

## Exit Criteria

- `cargo pgrx regress pg18` passes all tests including `v0101_explain_inference`
- `explain_inference()` on an inferred fact returns non-empty text in both LLM and fallback modes
- `explain_inference()` on a base fact returns `null`
- `explain_inference_jsonb()` returns a JSONB object with both expected keys
- `vacuum_explanation_cache()` removes rows older than the TTL and returns the count
- `POST /explain` HTTP endpoint returns a valid JSON response
