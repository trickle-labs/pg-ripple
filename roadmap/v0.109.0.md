# v0.109.0 — NS-RL Foundation: String Similarity Builtins + Orchestrator

**Theme**: Neuro-symbolic record linkage foundation.

> **In plain language:** pg_ripple has always had the building blocks for entity resolution —
> OWL axioms for deterministic matches, pgvector for fuzzy candidates, SHACL for validation,
> and `owl:sameAs` for canonicalization. This release wires those pieces together into a
> first-class, one-call API (`resolve_entities()`) and adds the string similarity SPARQL
> functions that were the single biggest gap for writing symbolic matching rules.
>
> **Foundation document**: [plans/neuro-symbolic-record-linkage.md](../plans/neuro-symbolic-record-linkage.md)
> §15.1 (gaps: string similarity builtins, orchestrator function) and §7 (synergies 1–7).
>
> **Effort estimate: 2–3 person-weeks**

## Prerequisites

- v0.87.0 (confidence) and v0.49.0 (`suggest_sameas`) available.
- v0.31.0 `owl:sameAs` canonicalization (union-find) available.
- Independent of v0.100.0–v0.108.0 (Expert System arc) — can ship in parallel.
- `pg_trgm` and `fuzzystrmatch` PostgreSQL extensions must be available in the database (checked at `_PG_init`).

## Deliverables

### String Similarity SPARQL Builtins

- [ ] `pg:trigram_similarity(?a, ?b) → xsd:double` — wraps `similarity(a, b)` from `pg_trgm`
- [ ] `pg:levenshtein(?a, ?b) → xsd:integer` — wraps `levenshtein(a, b)` from `fuzzystrmatch`
- [ ] `pg:levenshtein_less_equal(?a, ?b, ?max) → xsd:integer` — bounded Levenshtein (stops at `max`)
- [ ] `pg:soundex(?s) → xsd:string` — wraps `soundex(s)` from `fuzzystrmatch`
- [ ] `pg:metaphone(?s, ?maxlen) → xsd:string` — wraps `metaphone(s, maxlen)` from `fuzzystrmatch`
- [ ] `pg:jaro_winkler(?a, ?b) → xsd:double` — wraps `jarowinkler(a, b)` from `fuzzystrmatch` (PG17+)
- [ ] All six functions available as Datalog built-in predicates with the same `pg:` prefix
- [ ] `pg_trgm` and `fuzzystrmatch` extension availability checked at `_PG_init`; informational GUC `pg_ripple.string_similarity_extensions_ok BOOL` (read-only, set at startup)
- [ ] SPARQL FILTER tests for all six functions in `tests/pg_regress/sql/v0109_string_similarity.sql`

### High-Level `resolve_entities()` Orchestrator

- [ ] `pg_ripple.resolve_entities(source_graph TEXT, target_graph TEXT, options JSONB DEFAULT '{}') → JSONB`
  - Stage 1: symbolic blocking — if `blocking_rules` is set, loads and runs that Datalog rule set (e.g. from `er_blocking_template('email')`); by default runs `infer('owl-rl')` restricted to predicates declared `owl:InverseFunctionalProperty`. Blocking rules must derive `owl:sameAs` triples directly (head predicate = `owl:sameAs`) to be consumed by stage 4.
  - Stage 2: embedding candidates — calls `suggest_sameas(threshold, k)` if embeddings exist for the source graph
  - Stage 3: SHACL validation gate — checks proposed `owl:sameAs` pairs against enabled shapes; rejects violating pairs
  - Stage 4: canonicalization — runs union-find and rewrites aliases
  - Stage 5: RDF-star provenance — annotates each asserted `owl:sameAs` triple with `ex:confidence`, `ex:source`, `ex:resolvedAt`
  - Options: `blocking_rules TEXT` (Datalog rule set name for custom blocking — the rule set must derive `owl:sameAs` triples; use `load_rules(er_blocking_template('email'), 'er_email')` to create such a set), `confidence_threshold FLOAT8 DEFAULT 0.85`, `dry_run BOOLEAN DEFAULT false`, `max_candidates INT DEFAULT 10000`
  - `dry_run` returns candidate pairs without writing; result JSONB: `{"candidates": N, "symbolic": M, "neural": K, "would_assert": P, "blocked_by_shacl": Q}`
  - Non-dry-run result: `{"asserted": N, "blocked": M, "canonicalized": K, "duration_ms": T}`
- [ ] `pg_ripple.sameas_apply_rate_limit` GUC (INT, default: 1000): maximum `owl:sameAs` triples a single `resolve_entities()` call may assert — exceeding limit raises PT0460 (`entity resolution rate limit exceeded: %d of %d allowed sameAs assertions`)
- [ ] PT0460 entry in error catalog

### Built-in ER Blocking Templates

- [ ] `pg_ripple.er_blocking_templates() → TABLE(name TEXT, description TEXT, rule TEXT)` returning three templates:
  - `'email'` — `?x owl:sameAs ?y :- ?x schema:email ?e, ?y schema:email ?e, ?x != ?y .`
  - `'postal_name'` — `candidate(?x, ?y) :- ?x schema:postalCode ?z, ?y schema:postalCode ?z, ?x rdf:type ?c, ?y rdf:type ?c, ?x != ?y .`
  - `'name_prefix'` — blocking candidate on shared first 4 characters of `schema:name` via a `pg:levenshtein` Datalog predicate
- [ ] `pg_ripple.er_blocking_template(name TEXT) → TEXT` convenience function returning just the rule text
