## Deliverables

### M16-01 — ER Monitoring Data Retention

**Problem:** The `er_unresolved_entities`, `er_cluster_sizes`, and
`er_resolution_dashboard` tables grow without bound, causing bloat in long-running
deployments.

**Solution:** A GUC-controlled retention window and a prune function wired into
the background merge worker:

```sql
-- New GUC
pg_ripple.er_monitoring_retention_days INTEGER DEFAULT 30 RANGE [1, 3650]

-- New prune function
CREATE OR REPLACE FUNCTION _pg_ripple.er_monitoring_prune()
RETURNS void LANGUAGE sql AS $$
  DELETE FROM _pg_ripple.er_unresolved_entities
  WHERE recorded_at < now() - (current_setting('pg_ripple.er_monitoring_retention_days')::int
                                || ' days')::interval;
  DELETE FROM _pg_ripple.er_cluster_sizes
  WHERE recorded_at < now() - (current_setting('pg_ripple.er_monitoring_retention_days')::int
                                || ' days')::interval;
  DELETE FROM _pg_ripple.er_resolution_dashboard
  WHERE recorded_at < now() - (current_setting('pg_ripple.er_monitoring_retention_days')::int
                                || ' days')::interval;
$$;
```

The merge background worker calls `_pg_ripple.er_monitoring_prune()` once per
merge cycle (typically every 60 seconds) when `er_monitoring_vacuum` mode is active.

### M16-05 — Rule Explanation Cache Invalidation

**Problem:** `_pg_ripple.rule_explanations` could serve stale cached explanations
after `update_rule()` or `store_rules()` modified the active rule set.

**Solution:** A monotonically-increasing version stamp:

```sql
ALTER TABLE _pg_ripple.rule_explanations
  ADD COLUMN rule_version_stamp BIGINT NOT NULL DEFAULT 0;
```

`update_rule()` and `store_rules()` both call:
```sql
UPDATE _pg_ripple.rule_explanations
SET rule_version_stamp = rule_version_stamp + 1;
```

Query paths check the current stamp before serving a cached explanation:
```sql
SELECT explanation FROM _pg_ripple.rule_explanations
WHERE rule_id = $1
  AND rule_version_stamp = (SELECT max(rule_version_stamp) FROM _pg_ripple.rule_explanations)
```

### M16-07 — Proof-Tree Resource Guards

**New GUCs:**

| GUC | Type | Default | Range | Error code on overflow |
|-----|------|---------|-------|------------------------|
| `pg_ripple.proof_tree_max_depth` | `INT` | `64` | `1–1000` | `PT0480` |
| `pg_ripple.proof_tree_max_nodes` | `INT` | `10000` | `1–1000000` | `PT0481` |

Implementation in `src/prov.rs::assemble_proof_tree()`:

```rust
fn assemble_proof_tree(
    root: &Term,
    depth: usize,
    node_count: &mut usize,
) -> Result<ProofNode, PgError> {
    if depth > max_depth_guc() as usize {
        return Err(PgError::new("PT0480", "proof tree depth limit exceeded"));
    }
    *node_count += 1;
    if *node_count > max_nodes_guc() as usize {
        return Err(PgError::new("PT0481", "proof tree node count limit exceeded"));
    }
    // ...
}
```

Documentation in `docs/src/security.md` — "Resource Limits" section:
```markdown
### Proof Tree Limits

| GUC | Default | Purpose |
|-----|---------|---------|
| `pg_ripple.proof_tree_max_depth` | 64 | Prevent stack overflow on deeply recursive proofs |
| `pg_ripple.proof_tree_max_nodes` | 10000 | Limit memory usage for wide proof trees |
```

### M16-11 — Bidi Relay Overflow Policy GUC

**New GUC:**

| GUC | Type | Default | Options | Description |
|-----|------|---------|---------|-------------|
| `pg_ripple.bidi_relay_drop_policy` | `ENUM` | `'newest'` | `'newest'`, `'oldest'` | Which event to drop when relay buffer is full |

`'newest'` (default): drop the incoming event — existing buffered events are preserved.
`'oldest'`: drop the oldest buffered event to make room.

`'oldest'` is appropriate for deployments where downstream consumers can tolerate
replaying from a checkpoint but cannot tolerate missing recent events.

### M16-19 — Bounded Rule Explanation LRU

**Before:** `src/rule_explain.rs` used an unbounded `HashMap` for the explanation
cache, which could grow to hold millions of entries in a long-running deployment
with many distinct rule IDs.

**After:** Replaced with `lru::LruCache`:
```rust
use lru::LruCache;

static EXPLANATION_CACHE: Mutex<LruCache<i64, String>> =
    Mutex::new(LruCache::new(NonZeroUsize::new(1000).unwrap()));
```

The `lru` crate is added to `Cargo.toml` dependencies.

The cache size is controlled by:
```
pg_ripple.rule_explanation_cache_max_entries  INT  DEFAULT 1000  RANGE [1, 1000000]
```

Cache is re-initialised with the new capacity when the GUC is changed (requires
PostgreSQL reload).
