# v0.128.0 — JSON Mapping Relational Writeback (JSON-LD Reverse Mapping)

## Theme

Complete the `register_json_mapping` round-trip by adding a relational write-back
path (RDF → Relational). Changes to RDF triples can be automatically propagated
back to the source relational table via direct SQL calls or trigger-based
async queueing — closing the loop that began with `ingest_json()` (JSON → RDF)
and `export_json_node()` (RDF → JSON).

## Motivation

`register_json_mapping` already supports bidirectional JSON ↔ RDF conversion.
The missing half is the ability to write RDF graph changes back into the
originating relational table, enabling patterns such as:

- REST API receives a JSON payload → `ingest_json()` loads it as RDF →
  downstream SPARQL rules enrich or transform the data → `writeback_json_row()`
  writes the canonical form back to a `contacts` or `products` table.
- Any VP table change for a mapped predicate automatically enqueues a writeback
  event so the relational table stays in sync without application-level polling.

## Deliverables

### Schema

- [x] Add five new columns to `_pg_ripple.json_mappings`:
  - `writeback_table TEXT` — target relational table name
  - `writeback_schema TEXT NOT NULL DEFAULT 'public'` — target schema
  - `writeback_key_columns TEXT[] NOT NULL DEFAULT '{}'` — columns used as
    upsert primary key
  - `writeback_conflict_policy TEXT NOT NULL DEFAULT 'replace' CHECK
    (writeback_conflict_policy IN ('replace', 'skip', 'error'))` — behaviour on
    unique-key conflict
  - `writeback_enabled BOOLEAN NOT NULL DEFAULT false` — whether VP triggers
    are active
- [x] Add catalog table `_pg_ripple.json_writeback_queue`:
  ```sql
  CREATE TABLE _pg_ripple.json_writeback_queue (
      id            BIGSERIAL PRIMARY KEY,
      mapping_name  TEXT         NOT NULL
          REFERENCES _pg_ripple.json_mappings(name) ON DELETE CASCADE,
      subject_id    BIGINT       NOT NULL,
      operation     TEXT         NOT NULL CHECK (operation IN ('upsert', 'delete')),
      queued_at     TIMESTAMPTZ  NOT NULL DEFAULT now(),
      processed_at  TIMESTAMPTZ,
      error         TEXT
  );
  CREATE INDEX json_writeback_queue_pending_idx
      ON _pg_ripple.json_writeback_queue (mapping_name, queued_at)
      WHERE processed_at IS NULL;
  ```

### SQL API

- [x] Extend `pg_ripple.register_json_mapping()` with four optional parameters
  `writeback_table`, `writeback_schema`, `writeback_key_columns`,
  `writeback_conflict_policy` — all defaulting to NULL/defaults so existing
  call sites are unaffected
- [x] `pg_ripple.writeback_json_row(mapping TEXT, subject_iri TEXT) → BIGINT`
  — derives a JSON payload for the subject via the stored JSON-LD context,
  maps JSON keys to relational columns by querying `information_schema.columns`,
  and executes:
  - `INSERT … ON CONFLICT (key_cols) DO UPDATE SET …` when policy is `'replace'`
  - `INSERT … ON CONFLICT DO NOTHING` when policy is `'skip'`
  - raises `PT0551 ERROR: json mapping writeback conflict` when policy is `'error'`
  - raises `PT0550 ERROR: json mapping writeback target not configured` when
    `writeback_table` is NULL
  - returns rows affected
- [x] `pg_ripple.writeback_json_row_delete(mapping TEXT, subject_iri TEXT) → BIGINT`
  — decodes key-column values from VP tables and executes
  `DELETE FROM <target> WHERE <key_cols> = …`; returns rows affected
- [x] `pg_ripple.enable_json_writeback(mapping TEXT) → VOID`
  — validates `writeback_table` exists and `writeback_key_columns` is non-empty;
  sets `writeback_enabled = true`; installs `AFTER INSERT OR DELETE FOR EACH ROW`
  trigger `_pg_ripple.jwb_<mapping>_<predicate_id>` on each
  `_pg_ripple.vp_*_delta` table whose predicate IRI appears in the mapping context;
  each trigger calls `_pg_ripple.json_writeback_enqueue_fn()` which inserts a row
  into `json_writeback_queue`
- [x] `pg_ripple.disable_json_writeback(mapping TEXT) → VOID`
  — sets `writeback_enabled = false`; drops all `_pg_ripple.jwb_<mapping>_*`
  triggers
- [x] `pg_ripple.json_writeback_status() → TABLE(mapping_name TEXT, pending BIGINT,
  errors BIGINT, last_error TEXT, last_processed_at TIMESTAMPTZ)`
  — operational view over the queue grouped by mapping
- [x] `pg_ripple.json_writeback_batch_size` GUC (default 100, min 1, max 10000)
  — rows processed per background merge-worker tick; background worker drains
  the queue by calling `writeback_json_row()` / `writeback_json_row_delete()`
  in batches and updating `processed_at` / `error`

### HTTP Endpoints (`pg_ripple_http`)

- [x] `POST /json-mapping/{name}/writeback` — body `{"subject_iri": "…"}`; calls
  `writeback_json_row()` synchronously; returns `{"rows_affected": N}`;
  requires write-auth
- [x] `GET /json-mapping/{name}/writeback/status` — returns queue depth, error
  count, and `last_error` JSON; requires read-auth
