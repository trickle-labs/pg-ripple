# v0.84.0 Full — Assessment 13 Critical/High Findings (Source: PLAN_OVERALL_ASSESSMENT_13.md)

## Source

All findings sourced directly from `plans/PLAN_OVERALL_ASSESSMENT_13.md`,
Assessment #13 conducted against the v0.83.0 codebase.

## Area 1: Security

### S13-01 | HIGH | Effort: M
**Two `SECURITY DEFINER` functions lack inline justification**

`grep -rn "SECURITY DEFINER" src/ sql/` returns exactly 2 hits:
- `src/schema.rs:996`
- `sql/pg_ripple--0.55.0--0.56.0.sql:60`

Neither has a comment explaining why the privilege escalation is needed.

**Fix:** Add `-- SECURITY-JUSTIFY: <reason>` immediately after each
`SECURITY DEFINER`; update `check_no_security_definer.sh` to require the
marker rather than just reporting the occurrence.

### S13-02 | HIGH | Effort: M
**CI does not enforce the SQL injection check**

`scripts/check_no_string_format_in_sql.sh` exists but is not wired as a
required step in `.github/workflows/ci.yml`. 126 `format!()` patterns that
interpolate values into SQL strings exist across the codebase. The allowlist
(numeric pred_id/graph_id only) is correct but unenforced in CI.

**Fix:** Wire the script as a required CI step; audit all 126 occurrences
against the allowlist.

### S13-05 | HIGH | Effort: S
**`COMPATIBLE_EXTENSION_MIN` is `"0.79.0"` despite extension being at `0.83.0`**

`pg_ripple_http/src/main.rs:38` declares
`const COMPATIBLE_EXTENSION_MIN: &str = "0.79.0"`. The extension is at
`0.83.0`. Any breaking API change in v0.80–v0.83 is silently tolerated.

**Fix:** Raise `COMPATIBLE_EXTENSION_MIN` to `"0.84.0"` when bumping the
companion to `0.84.0`. Add `PG_RIPPLE_HTTP_STRICT_COMPAT=1` env var that
converts the mismatch warning to a startup error.

**Fix:** Bump to `:0.84.0`; add CI lint asserting the tag matches
`Cargo.toml`.

## Area 3: Tests

### T13-01 | HIGH | Effort: M
**Migration chain test coverage stops at v0.79.0**

`grep -nE "0\.8[0-9]" tests/test_migration_chain.sh` returns only line 383
(a header comment). The v0.80, v0.81, v0.82, v0.83 migrations apply but have
zero checkpoint assertions. A schema regression in any of the four newest
migrations passes CI silently.

**Fix:** Add checkpoint at:
- v0.80.0: CDC slot tables
- v0.81.0: `strict_dictionary` GUC, `dict_subxact` tracking
- v0.82.0: `_pg_ripple.merge_worker_status` table (verified by
  `sql/pg_ripple--0.81.0--0.82.0.sql:36`)
- v0.83.0: any new tables from that release
Total: approximately 30 lines of bash.

## Area 4: Code Quality

### Q13-01 | HIGH | Effort: M
**`src/gucs/registration.rs` is 2,032 lines — hardest file to navigate**

`find src -name "*.rs" -exec wc -l {} \;` returns 2,032 as the top result.
The file registers GUCs for every domain (SPARQL, storage, federation, Datalog,
security, observability) in one monolithic module.

**Fix:** Split into per-domain submodules:
- `src/gucs/registration/sparql.rs`
- `src/gucs/registration/storage.rs`
- `src/gucs/registration/federation.rs`
- `src/gucs/registration/datalog.rs`
- `src/gucs/registration/security.rs`
- `src/gucs/registration/observability.rs`

Keep public re-exports from `src/gucs/registration.rs` so no callers need
updating.

## Area 5: Correctness

### C13-01 | HIGH | Effort: M
**Nested `OPTIONAL` + `EXISTS` filter-pushdown regression**

`src/sparql/translate/{filter,left_join}` handles nested OPTIONAL blocks. A
regression introduced in v0.80–v0.83 may cause `OPTIONAL { ... FILTER(EXISTS
{ ... }) }` patterns to push the EXISTS filter outside the OPTIONAL, changing
query semantics.

**Fix:** Add a regression test in
`tests/pg_regress/sql/sparql_optional.sql`; fix the filter-pushdown logic if
the test fails.

## Area 6: Performance

### P13-01 | HIGH | Effort: M
**Plan cache double-parse on every cache-miss**

`src/sparql/plan_cache.rs` `cache_key()` currently calls
`spargebra::Query::parse(query_text)` to compute the canonical form for the
cache key hash. The caller then re-parses the same string. This means every
cache-miss incurs two full SPARQL parses.

**Fix:** Refactor `cache_key()` to accept a pre-parsed `&spargebra::Query`
(add `cache_key_from_query(q: &spargebra::Query) -> String` helper); update
callers in `execute.rs` and `plan.rs` to parse once and pass through.

## Area 7: Observability

### O13-01 | HIGH | Effort: M
**No `/health/ready` deep-check endpoint in `pg_ripple_http`**

Per CHANGELOG BUILD-TIME-FIELD-01 (v0.83.0), `/health` includes a
`build_time` field, but the response does not call SPI to verify PostgreSQL
connectivity or the extension version per request. Kubernetes readiness probes
using `/health` will pass even when the DB is unreachable.

**Fix:** Add a `/health/ready` route that runs
`SELECT 1 FROM pg_extension WHERE extname='pg_ripple'` with a 2-second
timeout; return `503 Service Unavailable` on failure. Keep `/health` as a fast
liveness probe.

## Area 9: Process & Scope (PROMPT-01, V084-01)

### PROMPT-01 | CRITICAL | Effort: — (process)
**Assessment prompt anchored on undelivered planned versions**

The Assessment #13 prompt (`overall_assesment_prompt.md`, lines 28–31) stated
that v0.84.0 had been delivered: "*v0.84.0: Uncertain Knowledge & Soft
Reasoning (probabilistic Datalog, fuzzy SPARQL filtering, trust scoring,
soft-constraint checking, linguistic entity resolution)*". At HEAD `142d8f2`,
the latest tag is `v0.83.0`; `Cargo.toml` is `0.83.0`. v0.84.0 was a
forward-looking planning artefact.

**Impact:** Auditors report findings against non-existent code, and the
assessment cannot be repeated reliably. This is a documentation-process risk
that recurs on every assessment cycle.

**Fix:** Establish a rule: assessment prompts are anchored at the latest
*tagged* release. Update `plans/overall_assesment_prompt.md` for Assessment
#13 to read "Conduct the assessment against `v0.83.0` (HEAD at time of
assessment)" and add a template note for future assessments:
"Scope = latest tag reachable via `git describe --tags --abbrev=0`."

### V084-01 | HIGH | Effort: — (scoping decision)
**v0.84.0 uncertain knowledge scope undecided at audit time**

v0.84.0 was scoped in `ROADMAP.md` as the "Uncertain Knowledge Engine" —
probabilistic Datalog with `@weight(FLOAT)` rule annotations, fuzzy SPARQL
filtering (`pg:fuzzy_match()`, `pg:token_set_ratio()`), trust scoring,
soft-constraint SHACL, and linguistic entity resolution. Estimated 4–8
person-weeks. No code existed at the time of Assessment #13.

Assessment #13 raised V084-01 as a HIGH finding: either commit to v0.84.0 as
planned (XL scope), split it across two releases, or defer past v1.0.0. Not
choosing was identified as the failure mode.

**Fix:** Decision taken in response to Assessment #13:
1. Rename the uncertain knowledge engine to **v0.87.0** (deferred, not cut).
2. Reserve **v0.84.0, v0.85.0, v0.86.0** for Assessment 13 remediation work.
3. Update `ROADMAP.md` with the new section structure and version assignments.
4. Update `plans/overall_assesment_prompt.md` per PROMPT-01.

This is the version that documents and closes that scoping decision.
