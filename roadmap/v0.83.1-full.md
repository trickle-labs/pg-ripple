# v0.83.1 — Assessment 13 Critical/High & Operational Remediation — Full Details

> Source: PLAN_OVERALL_ASSESSMENT_13.md  
> Status: Planned  
> Scope: Large

## Background

Assessment 13 (April 2026) identified 82 findings across 16 areas.  v0.83.1 addresses
the 13 findings classified as "must fix before v1.0.0" or rated Critical/High, plus
several closely related operational items that can be delivered in the same release.

The single Critical finding — PROMPT-01 (process risk from schema-drift between
`pg_ripple_http` and the extension) — is addressed directly by HTTP-01 / MF-B.

---

## Findings Addressed

### HTTP Companion Version Drift (HTTP-01 / MF-B) — Critical

**Problem**: `pg_ripple_http` is at v0.77.0 while the extension is at v0.83.0
(6 minor versions behind).  `COMPATIBLE_EXTENSION_MIN` in
`pg_ripple_http/src/main.rs:38` still reads `"0.79.0"`, meaning any extension
version from 0.79.0 through 0.83.0 passes the compatibility check even when it
should not.  New HTTP-layer features and routing changes introduced in v0.78–v0.83
are not reflected in the companion binary users are instructed to deploy.

**Fix**:
- Bump `version` in `pg_ripple_http/Cargo.toml` to `0.83.0`.
- Update `COMPATIBLE_EXTENSION_MIN` constant in `pg_ripple_http/src/main.rs` to
  `"0.83.0"`.
- Add a CI step that fails if the two version strings diverge beyond one minor
  version.

### Docker-Compose Image Tag (BUILD-01) — High

**Problem**: `docker-compose.yml` lines 23 and 40 reference
`ghcr.io/grove/pg-ripple:0.54.0` — 29 versions stale.  A developer running
`docker compose up` gets an out-of-date environment silently.

**Fix**:
- Update both image references to `ghcr.io/grove/pg-ripple:0.83.0`.
- Add a shell check (or `make`/`just` lint target) that compares the tag in
  `docker-compose.yml` against `Cargo.toml` version and fails if they differ.

---

### SECURITY DEFINER Inline Justification (S13-01) — High

**Problem**: `src/schema.rs:996` and `sql/pg_ripple--0.55.0--0.56.0.sql:60`
contain `SECURITY DEFINER` without an adjacent justification comment.
`scripts/check_no_security_definer.sh` reports these as violations but its
allow-list is not kept in sync with the code.

**Fix**:
- Add `-- SECURITY-JUSTIFY: <reason>` comment above each SECURITY DEFINER site.
- Rewrite `check_no_security_definer.sh` to enforce the comment pattern rather
  than an allow-list.
- Add the script to CI as a required lint step.

---

### SQL-String Format CI Gate (S13-02) — High

**Problem**: `scripts/check_no_string_format_in_sql.sh` exists but is not invoked
in CI.  Assessment 13 found 126 `format!(…)` calls that construct SQL strings.
While most are benign (table-name quoting via `format_ident!` equivalent), the gate
ensures no new interpolations slip in unreviewed.

**Fix**: Wire the script as a required step in the `lint` CI job.

---

### Migration Chain v0.80–v0.83 Checkpoints (T13-01) — High

**Problem**: `tests/test_migration_chain.sh` was last updated for v0.51.0–v0.79.0.
The v0.80–v0.83 migration scripts exist but are not exercised by the test harness.

**Fix**: Extend the harness with upgrade steps:
```
v0.51.0 → … → v0.79.0 (existing)
v0.79.0 → v0.80.0 → v0.81.0 → v0.82.0 → v0.83.0 (new)
```
Each step runs `ALTER EXTENSION pg_ripple UPDATE TO 'X.Y.Z'` and verifies that
`SELECT extversion FROM pg_extension WHERE extname = 'pg_ripple'` matches.

### Nested OPTIONAL+EXISTS Correctness (C13-01) — High

**Problem**: A SPARQL pattern of the form:

```sparql
OPTIONAL { ?s :p ?o . FILTER(EXISTS { ?o :q ?r }) }
```

can produce incorrect results when the EXISTS sub-pattern references a variable
bound inside the OPTIONAL block.  The current filter-pushdown pass moves the EXISTS
filter outside the OPTIONAL context, changing its semantics.

**Fix**:
- Add a regression test that encodes the exact failing pattern.
- Fix the algebra optimizer to detect when an EXISTS argument references variables
  that are optionally bound and to preserve the correct evaluation order.

### `gucs/registration.rs` Module Split (Q13-01) — High

**Problem**: `src/gucs/registration.rs` is 2 032 lines — the third-largest file in
the codebase.  Every GUC domain (SPARQL, storage, federation, Datalog, security,
observability) is interleaved in a single flat file, making diffs large and
navigation slow.

**Fix**: Split into six submodules under `src/gucs/registration/`:
- `sparql.rs` — SPARQL engine GUCs
- `storage.rs` — VP table and HTAP storage GUCs
- `federation.rs` — remote SERVICE endpoint GUCs
- `datalog.rs` — Datalog reasoning GUCs
- `security.rs` — SSRF, tenant, auth GUCs
- `observability.rs` — metrics, explain, log GUCs

The `registration/mod.rs` re-exports each domain's `register_gucs()` function and
calls them from `_PG_init`.

### `/health/ready` Deep-Check (O13-01) — High

**Problem**: `/health/ready` returns 200 after the HTTP server starts listening,
even if the PostgreSQL connection pool is not yet ready.  Kubernetes readiness probes
can route traffic before the extension is reachable.

**Fix**:
- Execute `SELECT extversion FROM pg_extension WHERE extname = 'pg_ripple'` during
  the ready check.
- Return 503 if the query fails or if the version is below `COMPATIBLE_EXTENSION_MIN`.
- Keep `/health/live` as a lightweight TCP/memory check only.

---

### Plan Cache Double-Parse (P13-01) — High

**Problem**: `plan_cache.rs::cache_key()` calls `spargebra::Query::parse()` to
produce a canonical string key.  At the call site the query has *already* been
parsed (the `Query` object exists).  This means every cache miss parses the query
text twice.

**Fix**: Change `cache_key` to accept `&spargebra::Query` and derive the key from
the already-parsed AST representation (e.g. `format!("{:?}", query)` or a
deterministic serialisation).  Thread the pre-parsed reference through all call
sites.

---

### `audit.toml` Expiry Documentation (S13-04) — Medium

**Problem**: Advisory ignores in `audit.toml` have no expiry dates or resolution
comments, making it impossible to tell whether a workaround is still applicable.

**Fix**: Add a `# Expires: YYYY-MM-DD` comment and a `# Resolution:` note to each
ignore block.  Document the `serde_cbor` removal investigation outcome.

---

### `PG_RIPPLE_HTTP_STRICT_COMPAT` Env Var (S13-05) — Medium

**Problem**: The COMPAT-01 extension version check emits a warning log but does not
prevent traffic from reaching an incompatible extension.  CI environments cannot
reliably assert that a bad version combination is caught.

**Fix**: When `PG_RIPPLE_HTTP_STRICT_COMPAT=1`, return HTTP 503 (Service
Unavailable) for all requests when the extension version is below
`COMPATIBLE_EXTENSION_MIN`, rather than just logging a warning.

---

### `justfile` Automation Recipes (BUILD-02) — Medium

**Problem**: Releasing a new version requires manual steps: bump Cargo.toml,
regenerate SBOM, regenerate OpenAPI spec, update docker-compose tag.  These are
error-prone and were the root cause of BUILD-01.

**Fix**: Add three `justfile` recipes:
- `bump-version NEW_VERSION` — updates Cargo.toml, docker-compose.yml,
  `COMPATIBLE_EXTENSION_MIN` in one atomic step.
- `regen-sbom` — runs `cargo cyclonedx` and commits `sbom.json`.
- `regen-openapi` — generates `docs/openapi/openapi.yaml` and verifies it is
  up to date.

---

### Migration-Chain CI Workflow (BUILD-03) — Medium

**Problem**: `tests/test_migration_chain.sh` is not invoked in any GitHub Actions
workflow.  It runs only if a developer remembers to run `just test-migration`.

**Fix**: Add a `migration-chain` job to `.github/workflows/ci.yml` that starts the
pgrx PG18 instance and runs `bash tests/test_migration_chain.sh`.
