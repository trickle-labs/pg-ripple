## Feature 5 — `owl:propertyChainAxiom` Inference

### Background

`owl:propertyChainAxiom` is an OWL 2 axiom that defines a property as the composition
of a chain of other properties. For example:

```turtle
:hasGrandparent owl:propertyChainAxiom ( :hasParent :hasParent ) .
```

This means: if `?x :hasParent ?y` and `?y :hasParent ?z`, then infer `?x :hasGrandparent ?z`.

The W3C OWL 2 RL profile (Table 9, rule prp-spo2) requires this entailment:
```
T(?p, owl:propertyChainAxiom, ?x),
LIST[?x, ?p1, ..., ?pn],
T(?u1, ?p1, ?u2), T(?u2, ?p2, ?u3), ..., T(?u(n-1), ?p(n-1), ?un)
⊨ T(?u1, ?p, ?un)
```

### Implementation

The chain is stored in an RDF list. The Datalog compiler materialises the chain
as a sequence of joins at rule-load time:

```sql
-- For owl:propertyChainAxiom (?p1 ?p2):
INSERT INTO _pg_ripple_rules VALUES (
  'prp_chain_<hash>',
  'chain_derived(?u1, ?u3) :- vp_p1(?u1, ?u2), vp_p2(?u2, ?u3)',
  'derived'
);
```

Variable-length chains (n > 2) are compiled recursively:
```datalog
% 3-hop chain: owl:propertyChainAxiom (?p1 ?p2 ?p3)
chain_step_1(?u1, ?u2) :- vp_p1(?u1, ?u2).
chain_step_2(?u1, ?u3) :- chain_step_1(?u1, ?u2), vp_p2(?u2, ?u3).
chain_derived(?u1, ?u4) :- chain_step_2(?u1, ?u3), vp_p3(?u3, ?u4).
```

### Cycle Safety

When a chain contains the derived property (e.g., transitive closure expressed
as a 2-hop chain of the same property), the stratifier detects the cycle and
applies tabling (memoisation) to prevent infinite recursion. A test in
`owl_rl_sameas.sql` verifies this:

```sql
-- This MUST terminate (not loop infinitely)
INSERT INTO triples VALUES
  ('<p>', 'owl:propertyChainAxiom', '<_list1>'),
  ('<_list1>', 'rdf:first', '<p>'),
  ('<_list1>', 'rdf:rest', '<_list2>'),
  ('<_list2>', 'rdf:first', '<p>'),
  ('<_list2>', 'rdf:rest', 'rdf:nil');
SELECT run_datalog_rules();  -- must complete
```

### LUBM Integration

The LUBM benchmark includes `ub:teacherOf` chains via `ub:advisor`/`ub:memberOf`.
The propertyChainAxiom test is:
```sparql

SELECT ?x ?y ?z WHERE {
  ?x ub:advisor ?y .
  ?y ub:teacherOf ?z .
  ?x owl:propertyChainAxiom ?z .  -- should be inferred
```

## Feature 6 — SERVICE Circuit Breaker

### Background

Federated SPARQL SERVICE queries can cause cascading failures when a remote
endpoint is slow or unreachable. Without a circuit breaker, all queries touching
that endpoint block for the full timeout (default 30s), exhausting connection
pool slots.

### State Machine

```
CLOSED → (failure_count >= threshold) → OPEN
OPEN   → (recovery_secs elapsed)      → HALF_OPEN
HALF_OPEN → (probe succeeds)          → CLOSED
HALF_OPEN → (probe fails)             → OPEN
```

### Schema

```sql
CREATE TABLE _pg_ripple.federation_circuit_state (
  endpoint_iri   TEXT PRIMARY KEY,
  state          TEXT NOT NULL DEFAULT 'closed'
                 CHECK (state IN ('closed', 'open', 'half_open')),
  last_failure_at TIMESTAMPTZ,
  failure_count  INT NOT NULL DEFAULT 0,
  last_success_at TIMESTAMPTZ
);
```

### GUCs

| GUC | Default | Description |
|-----|---------|-------------|
| `pg_ripple.federation_circuit_failure_threshold` | `5` | Failures before tripping |
| `pg_ripple.federation_circuit_recovery_secs` | `60` | Seconds before retry |

### Prometheus Gauge

```
pg_ripple_federation_circuit_state{endpoint="https://dbpedia.org/sparql"} 0
# 0 = closed (healthy), 1 = open (failing), 2 = half_open (probing)
```

### Behaviour When Circuit Is Open

When a SERVICE query targets an endpoint in `'open'` state, the query immediately
raises a SPARQL error:

```
SPARQL execution error: SERVICE <endpoint> circuit open (last failure: 2024-01-15T10:23:45Z, failures: 7/5)
```

This is consistent with how federated query engines handle unavailable endpoints
(fail fast rather than hang).

## Property Paths over RDF-star

### Background

Property path expressions like `?x foaf:knows+ ?z` use `WITH RECURSIVE` CTEs.
When the subject or object of a triple is itself a quoted triple `<< ?s ?p ?o >>`,
the standard VP table routing breaks because the encoded ID refers to a reified
triple, not a dictionary-encoded IRI.

### Fix

The SPARQL algebra compiler detects when a property path traversal could encounter
RDF-star nodes and routes the scan through `_pg_ripple.temporal_facts` (which
stores RDF-star statements) in addition to VP tables.

The union is expressed as:
```sql
WITH RECURSIVE path AS (
  SELECT s, o FROM vp_p WHERE s = $start
  UNION ALL
  SELECT t.subject, t.object FROM _pg_ripple.temporal_facts t
  JOIN path ON path.o = t.quoted_triple_id
  WHERE t.predicate = $predicate_id
)
SELECT DISTINCT o FROM path;
```

## Feature 10 — Schema-Aware NL→SPARQL

### Background

The `sparql_from_nl()` function (v0.96.0) sends a generic SPARQL grammar prompt
to the LLM. When vocabulary bundles are active (SKOS, DCTERMS, Schema.org, FOAF),
the LLM has no knowledge of the specific IRIs and labels in use, leading to
hallucinated property names.

### Implementation

When `pg_ripple.nl_sparql_include_bundles = on`, a grounding query fetches the
active bundle metadata:

```sql
-- Grounding query (conceptual)
SELECT DISTINCT
  dict_decode(s) AS entity_iri,
  dict_decode(o) AS label
FROM _pg_ripple.vp_{skos_prefLabel_id}
LIMIT 200;  -- capped to avoid token overflow
```

This is prepended to the LLM system prompt as a "vocabulary context" section:

```
Known vocabulary terms (use these IRIs exactly):
- <https://schema.org/Person> (schema:Person) — "A person"
- <https://schema.org/Organization> — "An organization"
- <http://xmlns.com/foaf/0.1/knows> (foaf:knows) — "knows"
...
```

Token cap: 2000 tokens for vocabulary context (configurable via
`pg_ripple.nl_sparql_bundle_context_tokens INT DEFAULT 2000`).
